## 1. 현재 상황 분석

### 1.1 현재 파이프라인

```
.글 소스 → 형태소분석기 → 구문분석기(AST) → IR변환기(3-주소 TAC) → PE생성기(x86-64)
```

### 1.2 현재 구현된 최적화

| 최적화 | 위치 | 효과 |
|--------|------|------|
| 피프홀 복사 제거 | `PE_피프홀최적화` | 불필요한 `명령_복사` 제거 |
| 죽은 저장 제거 | `PE_죽은저장제거` | 결과 미사용 명령 제거 (기본 블록 내) |
| 레지스터 핀 고정 | `PE_레지스터분석` | 상위 4개 빈도 변수 → rbx, r12-r14 |
| 인라이닝 분석 | `PE_인라이닝분석` | 유효 명령 4개 이하 함수 후보 식별 (미적용) |

### 1.3 현재 성능 대비 C(/O2)

| 벤치마크 | 배율 | 주요 병목 |
|----------|------|-----------|
| 재귀 피보나치(40) | 1.6x | 함수 프롤로그/에필로그 오버헤드 |
| 소수 체 | 2.6x | 루프+배열 접근 최적화 부족 |
| 제곱합 | 6.6x | 스택 왕복(load-op-store), SIMD 부재 |
| malloc/free | 1.04x | 외부 함수 호출 위주, 차이 없음 |
| 반복 피보나치 | 1.0x | 핀 고정이 이미 효과적 |

### 1.4 코드 생성 패턴의 핵심 문제

현재 PE생성기는 모든 IR 명령을 다음 패턴으로 변환한다:

```
PE_적재_r10(인자1)    →  mov r10, [rbp-오프셋1]
PE_적재_r11(인자2)    →  mov r11, [rbp-오프셋2]
인코딩_연산()          →  add r10, r11
PE_저장_r10(결과)     →  mov [rbp-오프셋3], r10
```

핀 고정된 변수만 레지스터에 상주하고, 나머지 모든 임시변수는 매번 스택을 왕복한다.
이것이 성능 격차의 근본 원인이다.

## A. SOV 인자 평가 순서 보장 최적화

### A.1 배경

글의 SOV(주어-목적어-동사) 어순에서 인자는 항상 동사(함수) 앞에 왼쪽에서 오른쪽 순서로 평가된다:

```글
값1을 값2를 값3으로 함수다.
```

이는 C의 함수 인자 평가 순서가 미정의(unspecified)인 것과 대조된다. C에서 `f(a(), b(), c())`의 a, b, c 평가 순서는 컴파일러 자유이므로, 최적화기가 순서를 바꿀 때 부수효과 간섭을 분석해야 한다.

### A.2 최적화 기법: 인자 파이프라이닝 (Argument Pipelining)

**핵심 아이디어**: 글에서 인자 평가 순서가 언어 수준에서 확정되므로, 인자를 평가하면서 동시에 목표 레지스터에 직접 배치할 수 있다.

현재 코드 (IR변환기.글 292-300행):
```
인자 순회 → 표현식변환 → 인자전달명령생성 → (PE생성기에서) 임시변수 → 스택 → 레지스터
```

최적화 후:
```
인자 순회 → 표현식변환 결과를 직접 목표 레지스터에 배치
```

**구체적 구현:**

1. **직접 레지스터 배치**: 인자가 단순 변수/상수이고, 해당 인자 이후의 인자들이 이 레지스터를 clobber하지 않으면, 표현식 결과를 즉시 rcx/rdx/r8/r9에 배치한다.

2. **역순 안전성 분석**: SOV 순서 보장 덕분에 역순 안전성 분석이 불필요하다. i번째 인자 평가 시 i+1...n번째 인자의 부수효과가 i번째 결과를 변경할 가능성을 검사할 필요 없이, 순서대로 평가하면 된다. C 컴파일러는 이 분석이 필수적이다.

3. **의존성 없는 인자의 병렬 평가 힌트**: 인자 간 데이터 의존성이 없으면(순수 표현식), CPU의 비순차 실행기가 병렬 처리할 수 있도록 독립적인 레지스터 체인으로 생성한다.

**예시:**
```글
(가 + 나)를 (다 * 라)를 함수다.
```

현재 생성 코드:
```asm
mov r10, [rbp-가]     ; 인자1 계산
mov r11, [rbp-나]
add r10, r11
mov [rbp-임시1], r10   ; 스택에 저장
mov r10, [rbp-다]     ; 인자2 계산
mov r11, [rbp-라]
imul rax, r11
mov [rbp-임시2], rax
mov rcx, [rbp-임시1]   ; 인자 배치
mov rdx, [rbp-임시2]
call 함수
```

최적화 후:
```asm
mov rcx, [rbp-가]     ; 인자1 → 직접 rcx로
mov r10, [rbp-나]
add rcx, r10
mov rdx, [rbp-다]     ; 인자2 → 직접 rdx로
mov r10, [rbp-라]
imul rdx, r10          ; rdx에서 직접 계산
call 함수
```

**C에서 불가능한 이유**: C 컴파일러는 `f(a(), b())`에서 `a()`가 `b()`에 영향을 줄 수 있으므로 순서를 함부로 정할 수 없다. 글은 순서가 확정이므로 인자 레지스터 할당을 앞당길 수 있다.

### A.3 기대 효과

| 항목 | 값 |
|------|-----|
| 성능 향상 | 함수 호출당 2-4개 mov 제거 → 호출 집약 코드에서 5-15% |
| 구현 복잡도 | PE_호출명령인코딩 수정 ~150줄 |
| 대상 약점 | 재귀 피보나치 1.6x → ~1.4x |

## B. 조사 기반 레지스터 할당

### B.1 배경

글의 조사(particle)는 인자의 문법적 역할을 명시한다:
- `을/를` (대격) — 직접 목적어, 주요 피연산자
- `에` (처소격) — 위치/대상
- `로/으로` (도구격) — 수단/방법
- `의` (속격) — 소유/멤버
- `은/는` (주격 보조사) — 주어/주제

### B.2 최적화 기법: 조사-레지스터 친화도 (Particle-Register Affinity)

**핵심 아이디어**: 조사별로 선호 레지스터를 지정하여, 함수 호출 규약과 자연스럽게 정렬시킨다.

Windows x64 ABI 인자 순서: rcx, rdx, r8, r9

글의 인자 순서는 조사로 결정된다. 만약 조사와 레지스터 매핑을 표준화하면:

| 조사 | 의미적 역할 | 매핑 레지스터 | 근거 |
|------|------------|--------------|------|
| 을/를 | 주요 피연산자 | rcx (1번 인자) | 가장 중요한 데이터 |
| 에 | 대상/위치 | rdx (2번 인자) | 목적지 |
| 로/으로 | 수단 | r8 (3번 인자) | 보조 정보 |
| (나머지) | 기타 | r9 (4번 인자) | 추가 정보 |

**그러나 이 기법에는 근본적 한계가 있다:**

현재 글의 인자 순서는 조사가 아닌 **출현 순서**로 결정된다. IR변환기.글의 함수호출 변환(282-306행)은 인자를 연결 리스트 순서대로 처리한다. 조사 정보는 AST에 보존되지만, IR 수준에서는 순번(인자전달명령생성의 `순번` 매개변수)으로만 표현된다.

### B.3 실현 가능한 변형: 조사 기반 인라인 최적화 힌트

조사를 레지스터에 직접 매핑하는 대신, 조사 정보를 **최적화 힌트**로 활용할 수 있다:

1. **`의` 조사 → 멤버 접근 체인 최적화**: `객체의 필드의 하위필드` 패턴에서 기본 주소를 레지스터에 유지하고 오프셋만 변경한다. 현재는 매 접근마다 기본 주소를 스택에서 다시 로드한다.

2. **`로/으로` 조사 → 형변환 인라이닝**: `정수로` 같은 형변환이 함수 호출 없이 인라인 변환됨을 조사 수준에서 확인하여, 불필요한 인자 전달을 제거한다.

3. **`을/를` 누적 패턴**: `합계를 ... 더하기다` 같은 패턴에서 `을/를`이 붙는 대상이 누적 변수임을 인식하고, 해당 변수를 누적기(accumulator) 레지스터에 고정한다.

### B.4 기대 효과

| 항목 | 값 |
|------|-----|
| 성능 향상 | 멤버 체인 접근에서 10-20%, 일반 코드에서 3-5% |
| 구현 복잡도 | AST에 조사 태그 보존 + IR 주석 확장 ~300줄 |
| 대상 약점 | 구조체 집약 코드, 멤버 접근 체인 |

### B.5 C에서 불가능한 이유

C에는 인자의 의미적 역할을 표현하는 문법 요소가 없다. `f(a, b, c)`에서 a, b, c의 역할은 위치로만 구분된다. 글의 조사는 의미적 역할이 문법에 내장되어 있어, 컴파일러가 데이터 흐름의 의도를 파악할 수 있다.

## C. 형태소 기반 분기 예측 최적화

### C.1 배경

한국어 어미(verb ending)는 제어 흐름을 명시적으로 인코딩한다:

| 어미 | 의미 | 제어 흐름 |
|------|------|-----------|
| `이면` / `면` | 조건 | 조건 분기 (if) |
| `동안` | 반복 | 루프 백엣지 (while) |
| `다` | 서술 종결 | 순차 실행 (fall-through) |
| `고` | 연결 | 순차 연결 (and-then) |

C에서는 `if`, `while`, `for` 등의 키워드가 제어 흐름을 표시하지만, 이들은 별도의 토큰이다.
글에서는 **표현식 자체의 어미가 제어 흐름을 결정**한다.

### C.2 최적화 기법: 어미 선행 분기 템플릿 (Ending-Driven Branch Template)

**핵심 아이디어**: 형태소분석기가 어미를 인식하는 순간, 파서를 거치지 않고도 분기 구조를 예측할 수 있다. 이를 코드 생성 단계에서 활용한다.

**기법 1: 조건분기 퓨전 (Condition-Branch Fusion)**

현재 생성 코드 (`이면` 처리):
```asm
; 조건식 계산 → r10
mov r10, [rbp-조건]     ; 조건값 로드
test r10, r10           ; 0 비교
jne L_참                ; 참이면 점프
jmp L_거짓              ; 거짓이면 점프
```

`이면` 어미가 비교 연산 직후에 항상 온다는 것을 알고 있으므로, 비교+분기를 퓨전할 수 있다:

최적화 후 (`cmp`+`이면` 퓨전):
```asm
; 비교 연산이 조건분기 직전이면 퓨전
mov r10, [rbp-왼쪽]
cmp r10, [rbp-오른쪽]   ; 또는 cmp r10, r11
jg L_참                 ; 비교 결과를 직접 분기에 사용
jmp L_거짓
```

**구현**: IR 수준에서 `비교 → 조건분기` 패턴을 감지한다. 비교 명령의 결과가 오직 바로 다음 조건분기에서만 사용되면, 비교와 분기를 하나의 기계어 시퀀스로 퓨전한다.

이미 PE생성기의 `PE_비교명령인코딩` (1693행)에서 `cmp r10, r11` + `setcc` + `movzx` 패턴으로 생성하고 있다. `setcc`+`movzx`로 0/1을 만든 뒤 다시 `test`+`jne`하는 대신, `cmp`+`jcc`로 직접 분기하면 3개 명령을 절약한다.

**기법 2: `동안` 루프 레이아웃 최적화**

`동안` 어미가 보이면 루프 백엣지임을 즉시 알 수 있다. 이를 활용하여:

- **루프 조건을 루프 하단에 배치**: 첫 반복에서만 조건 점프로 루프에 진입하고, 이후 반복은 하단의 조건 분기만 실행한다. 이렇게 하면 매 반복마다 무조건 점프(`jmp 조건라벨`)를 절약한다.

현재 (`동안` 루프, IR변환기.글 521-543행):
```
L_조건:
    조건 평가
    jne L_몸체
    jmp L_끝
L_몸체:
    몸체 코드
    jmp L_조건       ← 매 반복마다 무조건 점프
L_끝:
```

최적화 후 (하단 조건 배치):
```
    조건 평가           ← 첫 반복 진입 검사
    je L_끝
L_몸체:
    몸체 코드
    조건 평가           ← 하단에서 직접 분기
    jne L_몸체
L_끝:
```

### C.3 기대 효과

| 항목 | 값 |
|------|-----|
| 조건분기 퓨전 | 조건문당 3개 명령 절약 → 분기 집약 코드에서 10-20% |
| 루프 하단 배치 | 반복당 1개 `jmp` 절약 → 타이트 루프에서 5-15% |
| 구현 복잡도 | 조건분기 퓨전 ~200줄, 루프 레이아웃 ~150줄 |
| 대상 약점 | 소수 체 2.6x → ~2.0x, 제곱합 6.6x → ~5.5x |

### C.4 C에서도 가능한가?

조건분기 퓨전과 루프 레이아웃은 C 컴파일러도 수행한다. 그러나 글에서는 **어미 수준에서 제어 흐름이 확정**되므로, 파서 단계에서 이미 분기 템플릿을 준비할 수 있다. C 컴파일러는 AST를 완전히 구성한 후에야 이 분석이 가능하다.

글 고유의 이점은 **증분 코드 생성(incremental code generation)**에 있다: 형태소분석 → 파싱 → IR 변환 → 코드 생성을 스트리밍 방식으로 처리할 때, `이면` 어미를 만나는 즉시 분기 템플릿을 emit할 수 있다. 이는 메모리 사용량 감소와 컴파일 속도 향상으로 이어진다.

## D. 복합어 패턴 인식 최적화

### D.1 배경

글의 형태소분석기는 복합어를 분리한다:
- `명령종류풀[번호]` → 배열 `명령종류풀`의 `번호`번째 원소
- `임시변수크기풀[인덱스]` → 배열 `임시변수크기풀`의 `인덱스`번째 원소

중간표현.글(99-108행)에서 볼 수 있듯, SoA 패턴의 배열들은 매우 자주 같은 인덱스로 접근된다:

```글
정수 종류 = 명령종류풀[인덱스].
정수 결과 = 명령결과풀[인덱스].
정수 인자1 = 명령인자1풀[인덱스].
정수 인자2 = 명령인자2풀[인덱스].
```

### D.2 최적화 기법: SoA 접근 패턴 인식 (SoA Access Pattern Recognition)

**핵심 아이디어**: 동일한 인덱스로 여러 SoA 배열에 접근하는 패턴을 IR 수준에서 인식하고, 인덱스 계산을 공유한다.

현재 생성 코드:
```asm
; 명령종류풀[인덱스]
mov r10, [rbp-인덱스]       ; 인덱스 로드
imul r11, r10, 8             ; 오프셋 = 인덱스 * 8
lea r10, [rip+명령종류풀]
mov r10, [r10+r11]           ; 값 로드
mov [rbp-종류], r10

; 명령결과풀[인덱스]        ← 인덱스 다시 로드!
mov r10, [rbp-인덱스]
imul r11, r10, 8
lea r10, [rip+명령결과풀]
mov r10, [r10+r11]
mov [rbp-결과], r10
```

최적화 후:
```asm
; 인덱스를 한 번만 계산하여 레지스터에 유지
mov rbx, [rbp-인덱스]       ; 인덱스 → rbx (핀 고정)
shl rbx, 3                   ; 오프셋 = 인덱스 * 8

lea r10, [rip+명령종류풀]
mov r10, [r10+rbx]
mov [rbp-종류], r10

lea r10, [rip+명령결과풀]   ; 인덱스 재계산 없음
mov r10, [r10+rbx]
mov [rbp-결과], r10
```

**구현 방법:**

1. IR 수준에서 `명령_원소적재` 명령들을 스캔한다.
2. 연속된 원소적재 명령들이 동일한 인덱스 임시변수를 사용하면, 첫 번째에서 계산한 오프셋을 레지스터에 유지한다.
3. 기존 핀 고정 시스템을 확장하여 "인덱스 오프셋 임시 핀"을 추가한다.

### D.3 더 고급: 복합어 의미 분석

한글 복합어의 구조적 패턴을 활용:
- `명령종류풀`, `명령결과풀`, `명령인자1풀` → 접두사 `명령`이 동일
- 이는 논리적으로 같은 "구조체"의 다른 필드 접근

이 패턴을 인식하면:
- 접두사가 같은 배열들을 하나의 논리적 구조체 그룹으로 묶는다.
- 동일 인덱스 접근을 구조체 필드 접근처럼 최적화한다.
- 기본 주소 + 고정 오프셋 패턴으로 변환한다.

### D.4 기대 효과

| 항목 | 값 |
|------|-----|
| 성능 향상 | SoA 집약 코드에서 15-25%, 일반 코드에서 5-10% |
| 구현 복잡도 | IR 패턴 매칭 + 코드 생성 수정 ~400줄 |
| 대상 약점 | 소수 체 2.6x → ~2.2x (배열 집약) |

### D.5 글 고유성

이 최적화가 글에 특히 유효한 이유:
- 글의 `묶음` 정의 제약(전역 묶음 변수 불가)으로 인해 SoA 패턴이 강제된다.
- 한글 복합어의 접두사 공유 패턴이 관련 배열의 의미적 그룹핑 정보를 제공한다.
- C에서는 구조체를 직접 쓰면 되므로 SoA 패턴이 선택적이지만, 글에서는 필수적이다.

## E. 제곱합 6.6배 격차 해소

### E.1 문제 분석

제곱합 루프:
```글
(번호 < 최대)동안 {
    합계 = 합계 + 번호 * 번호.
    번호++다.
}
```

C(/O2) 생성 코드 (핵심 루프):
```asm
.loop:
    imul eax, ecx, ecx    ; 번호 * 번호 → eax
    add edx, eax          ; 합계 += eax
    inc ecx               ; 번호++
    cmp ecx, esi          ; 번호 < 최대
    jl .loop
```
5개 명령, 모든 변수가 레지스터에 상주.

글 현재 생성 코드 (추정):
```asm
.loop:
    mov r10, [rbp-번호]        ; 번호 로드
    mov r11, [rbp-번호]        ; 번호 로드 (다시!)
    imul rax, r11              ; 번호 * 번호
    mov [rbp-임시1], rax       ; 임시 저장
    mov r10, [rbp-합계]        ; 합계 로드
    mov r11, [rbp-임시1]       ; 임시 로드
    add r10, r11               ; 합계 + 임시
    mov [rbp-합계], r10        ; 합계 저장
    mov r10, [rbp-번호]        ; 번호 로드 (또!)
    inc r10
    mov [rbp-번호], r10        ; 번호 저장
    mov r10, [rbp-번호]        ; 조건 비교용 로드
    mov r11, [rbp-최대]
    cmp r10, r11
    jl .loop
```
~15개 명령, 매번 스택 왕복.

### E.2 최적화 기법 1: 루프 내 누적 변수 감지 (Loop Accumulator Detection)

**핵심 아이디어**: `합계 = 합계 + 표현식` 패턴을 IR 수준에서 감지하고, `합계`를 레지스터에 고정한다.

이는 이미 `PE_레지스터분석`의 핀 고정 시스템이 하는 일이다. 그러나 현재 시스템의 한계:

1. **빈도 기반만 사용**: 루프 가중치 없이 단순 사용 빈도만 센다 (1287-1304행).
2. **루프 감지 부재**: 주석에 "가장 간단한 접근: 그냥 모든 읽기 사용 빈도를 센다"라고 명시.
3. **최대 4개 제한**: rbx, r12-r14만 사용.

**개선안:**

1. **루프 가중치 도입**: 이동(jmp) 명령의 대상이 현재 위치 이전이면 루프 백엣지. 이 범위 내의 사용 빈도에 10배 가중치를 부여한다.

2. **핀 고정 확장**: r15도 callee-saved이므로 최대 5개로 확장. 또한 루프 내에서만 살아있는 임시변수도 핀 대상에 포함한다 (현재는 로컬/매개변수만).

3. **누적 패턴 특화**: `X = X op Y` 패턴을 감지하면, X를 누적 레지스터(rax 선호)에 고정하고, 연산을 메모리 왕복 없이 레지스터에서 직접 수행한다.

### E.3 최적화 기법 2: 스택 왕복 제거 (Stack Round-Trip Elimination)

**핵심 아이디어**: 임시변수의 수명이 한 IR 명령 내에서 끝나면 (정의 직후 사용, 이후 미사용), 스택에 저장하지 않고 레지스터에만 유지한다.

**구현:**

PE생성기에서 각 IR 명령 인코딩 시:
1. 결과 임시변수가 "단명(ephemeral)"인지 판단: 바로 다음 명령에서만 사용되고 이후 사용되지 않음.
2. 단명이면 `PE_저장_r10(결과)` 호출을 건너뛴다.
3. 다음 명령의 `PE_적재_r10(인자)` 시 "이전 명령의 결과가 아직 r10에 있음"을 인식하고 로드를 건너뛴다.

이를 위한 간단한 상태 머신:
```
PE_r10_현재임시 = -1   (현재 r10에 어떤 임시변수가 있는지)
PE_r11_현재임시 = -1
PE_rax_현재임시 = -1
```

`PE_저장_r10(임시)` 호출 시 임시가 단명이면 메모리 저장을 생략하고 `PE_r10_현재임시 = 임시`만 설정.
`PE_적재_r10(임시)` 호출 시 `PE_r10_현재임시 == 임시`이면 메모리 로드를 생략.

### E.4 최적화 기법 3: 강도 감소 (Strength Reduction)

`번호 * 번호` 패턴에서 동일 변수의 제곱은 `imul reg, reg`로 직접 생성할 수 있다. 현재는 두 피연산자를 별도 레지스터에 로드한 후 곱하지만, 동일 변수면 한 번만 로드한다.

### E.5 종합 기대 효과 (제곱합)

| 최적화 | 제거되는 명령 수 (루프당) | 누적 배율 |
|--------|--------------------------|-----------|
| 현재 | 0 | 6.6x |
| 루프 가중 핀 고정 | 3-4 (변수 로드/저장) | ~4.0x |
| 스택 왕복 제거 | 4-6 (임시변수 왕복) | ~2.5x |
| 누적 레지스터 | 2 (합계 로드/저장) | ~2.0x |
| 조건분기 퓨전 | 3 (setcc+movzx+test) | ~1.7x |
| 루프 하단 배치 | 1 (jmp) | ~1.5x |

## F. 함수 프롤로그/에필로그 최적화

### F.1 문제 분석

현재 모든 함수의 프롤로그 (PE_함수인코딩, 2301-2316행):
```asm
push rbx          ; (핀된 경우)
push r12          ; (핀된 경우)
push rbp
mov rbp, rsp
sub rsp, N        ; N = 프레임 크기 (최소 40바이트)
mov [rbp-8], rcx  ; 매개변수 저장
mov [rbp-16], rdx ; ...
```

에필로그:
```asm
leave             ; mov rsp, rbp; pop rbp
pop r12
pop rbx
ret
```

재귀 피보나치(40)에서 이 시퀀스가 ~2.1억 번 실행된다.

### F.2 최적화 기법 1: 리프 함수 감지 (Leaf Function Detection)

**정의**: 리프 함수는 다른 함수를 호출하지 않는 함수이다.

**리프 함수 최적화:**
- `push rbp` / `mov rbp, rsp` / `leave` 생략 → rsp-relative 주소 사용
- callee-saved 레지스터 push/pop 불필요 (호출이 없으므로 clobber 없음)
- 섀도 스페이스(32바이트) 불필요

```asm
; 리프 함수 프롤로그
sub rsp, 16       ; 로컬 변수만큼만

; 리프 함수 에필로그
add rsp, 16
ret
```

**구현**: `PE_프레임계산`에서 이미 호출 명령 수를 스캔한다 (1074-1083행). 호출이 0이면 리프 함수로 표시.

### F.3 최적화 기법 2: 프레임 크기 축소

현재 `PE_프레임계산`(1046행)은 모든 임시변수에 8바이트 슬롯을 할당한다. 그러나:

1. **핀 고정된 변수는 스택 슬롯 불필요**: 핀 고정 변수는 레지스터에만 살고, 스택 슬롯을 읽지 않는다 (초기 적재 후). 슬롯을 할당하되 크기를 0으로 줄일 수 있다.

2. **단명 임시변수 슬롯 제거**: E.3의 스택 왕복 제거와 연계하여, 단명 임시변수에 스택 슬롯을 할당하지 않는다.

3. **임시변수 수명 분석 기반 슬롯 재사용**: 수명이 겹치지 않는 임시변수들은 같은 스택 슬롯을 공유할 수 있다.

**재귀 피보나치의 경우:**
- 현재 프레임: ~144바이트 (`sub rsp, 144`)
- 실제 필요: 매개변수 1개(8B) + 로컬 2개(16B) + 섀도(32B) = 56바이트 (정렬 후 64바이트)
- 프레임 크기가 절반 이상 줄면, 재귀 호출의 스택 캐시 적중률이 높아진다.

### F.4 최적화 기법 3: 꼬리 호출 최적화 (Tail Call Optimization)

글의 SOV 구조에서 꼬리 호출 패턴:
```글
반환 값을 함수다.    (* 반환 + 호출이 합쳐진 형태 *)
```

IR에서 `명령_호출` 직후 `명령_반환`이 오고, 반환값이 호출 결과와 동일하면 꼬리 호출이다.

꼬리 호출 최적화:
```asm
; 꼬리 호출 전
leave
pop rbx        ; callee-saved 복원
jmp 함수       ; call+ret 대신 jmp
```

**피보나치에는 직접 적용 불가** (이중 재귀이므로), 그러나 반복적 재귀 패턴에는 유효하다.

| 기법 | 효과 | 구현량 |
|------|------|--------|
| 리프 함수 감지 | 리프 함수 호출당 ~8바이트 코드 절약 | ~100줄 |
| 프레임 축소 | 재귀 호출 시 캐시 효율 10-20% 향상 | ~200줄 |
| 슬롯 재사용 | 프레임 30-50% 축소 | ~300줄 |
| 꼬리 호출 | 꼬리 재귀에서 스택 오버플로 방지 | ~150줄 |
| **종합** | 재귀 피보나치 1.6x → ~1.3x | |

### 예상 최종 성능

Phase 1-2 구현 후:

| 벤치마크 | 현재 | 예상 | 목표 |
|----------|------|------|------|
| 재귀 피보나치 | 1.6x | 1.2-1.3x | < 1.5x |
| 소수 체 | 2.6x | 1.5-1.8x | < 2.0x |
| 제곱합 | 6.6x | 1.8-2.5x | < 2.0x (SIMD 제외) |

참고: SIMD 벡터화 없이 제곱합을 C(/O2) 수준까지 끌어올리는 것은 불가능하다. C 컴파일러의 자동 벡터화는 4-8배 병렬 처리를 수행하기 때문이다. 스칼라 코드 기준으로 2.0x 이내가 현실적 목표이다.

## 부록: 글 언어 고유 최적화의 학술적 의의

### 교착어 문법이 컴파일러 최적화에 주는 이점

1. **명시적 의미역 표시**: 조사가 인자의 역할을 문법적으로 확정하므로, 데이터 흐름 분석에 추가 의미 정보를 제공한다.

2. **확정적 평가 순서**: SOV 구조의 좌→우 평가 보장이 인자 간 간섭 분석을 불필요하게 만든다.

3. **형태소에 인코딩된 제어 흐름**: 어미가 제어 흐름을 결정하므로, 렉서 수준에서 제어 흐름 그래프의 뼈대를 예측할 수 있다.

4. **복합어의 구조적 정보**: 한글 복합어의 접두사/접미사 패턴이 변수/배열의 논리적 그룹핑 정보를 제공한다.

이러한 특성은 SVO(주어-동사-목적어) 언어 기반인 C/Rust/Go의 컴파일러에서는 구현할 수 없거나, 별도의 분석 패스가 필요한 것들이다. 글 컴파일러는 이러한 문법적 정보를 "무료로" 얻을 수 있다는 점에서 고유한 최적화 기회를 가진다.
