# 글 언어 독립 컴파일러 종합 로드맵

## 최종 목표

C 언어처럼 **완전히 자체적이고 독립적인 프로그래밍 언어**를 구축한다.

```
최종 파이프라인:
  .글 소스 → [글cc] → 네이티브 실행파일 (.exe / ELF)
                ↑
          글 언어로 작성된 컴파일러
          외부 도구 의존성 없음
```

### 현재 상태 (2026-03-17)

```
.글 → 형태소분석 → 구문분석 → AST → C 코드 → MSVC(cl.exe) → .exe
                                  → LLVM IR → clang/llc   → .exe (실험적)
```

- **자체호스팅 완료**: 컴파일러 100% 글 언어 (7개 파일, ~4100줄)
- 고정점 도달: Stage1 == Stage2 SHA256 일치
- C 백엔드: 완성도 높음 (대부분 기능 동작)
- LLVM 백엔드: 기초 구현 (~1040줄), 구조체/배열 미지원
- **현재 의존성**: MSVC(cl.exe) — C 트랜스파일 방식
- **한글 전용화 진행중**: 사용자 코드에서 영어 함수명 제거 목표

## P3: 중간표현(IR) 설계 및 구현 — 미시작

### P3.1: IR 설계 원칙

현재 AST에서 바로 코드를 생성하면 백엔드마다 중복 로직이 발생한다.
IR 계층을 도입하여 **프론트엔드(파서)와 백엔드(코드생성)를 분리**한다.

```
현재:  AST → C 코드생성기
       AST → LLVM 생성기 (중복 로직)

목표:  AST → 글IR → C 백엔드
                   → x86-64 백엔드
                   → (미래) ARM 백엔드
```

### P3.2: 글IR 명세

**3-주소 코드(TAC) 기반**, SSA는 초기에는 미적용.

```
글IR 명령어 종류:
─────────────────────────────────────────────
산술:     더하기, 빼기, 곱하기, 나누기, 나머지
비교:     같다, 다르다, 크다, 작다, 크거나같다, 작거나같다
논리:     그리고, 또는, 아님
메모리:   적재(load), 저장(store), 주소(addressof)
제어:     분기(branch), 조건분기(cbranch), 이동(jump), 반환(return)
호출:     호출(call), 인자전달(arg), 반환값받기(retval)
변환:     형변환(cast), 확장(extend), 축소(truncate)
구조체:   필드적재(field_load), 필드저장(field_store)
배열:     원소적재(elem_load), 원소저장(elem_store)
기타:     없음(nop), 레이블(label), 할당(alloca)
```

## 기술적 결정 사항

### 왜 LLVM을 사용하지 않는가?

현재 `llvm생성기.c`가 있지만 다음 이유로 자체 백엔드를 선택:

1. **LLVM 자체가 거대한 의존성** — LLVM은 수백 MB의 라이브러리
2. **목표가 완전한 독립** — C 언어처럼 자체적인 도구 체인
3. **학습/제어 가능성** — 모든 단계를 이해하고 제어
4. **LLVM IR 생성기는 유지** — 개발 과정에서 참조/검증 용도로 활용

### 왜 IR이 필요한가?

1. **백엔드 독립성** — 하나의 IR로 여러 백엔드 지원
2. **최적화 기회** — IR 수준에서 플랫폼 무관한 최적화 가능
3. **디버깅 용이** — IR을 텍스트로 덤프하여 검증
4. **자체호스팅 단순화** — 프론트엔드와 백엔드를 독립적으로 포팅

### 왜 P5를 P3/P4보다 먼저 완료했는가?

C 트랜스파일 백엔드를 유지하면서 자체호스팅을 먼저 달성하는 전략을 택했다:
- 컴파일러 소스를 글로 작성하는 것이 **사용자 가치**가 가장 높음
- IR/네이티브 백엔드는 **C 독립**을 위한 장기 목표
- 자체호스팅 완료 후 한글 전용화를 진행하여 사용자 경험을 우선 개선
