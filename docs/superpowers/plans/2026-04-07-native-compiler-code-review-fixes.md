- [ ] **Step 2: 긴정수 변수로 변경 및 imm64 인코딩 추가**

`정수 값`을 `긴정수 값`으로 변경하고, 상위 32비트가 있으면 `MOV r64, imm64` (REX.W + B8+r, 10바이트)로 인코딩:

```글
    (종류 == 명령_상수정수)이면 {
        긴정수 값은 명령정수값풀[명령인덱스].
        정수 상위 = 값 >> 32.
        정수 하위 = 값 & 4294967295.
        (* 상위 32비트가 있거나 부호확장 시 문제되는 경우 imm64 사용 *)
        (상위 != 0) + ((상위 == -1) * (하위 >= 0)) > 0이면 {
            (* MOV r10, imm64: 49 BA [8바이트] *)
            임시변수종류풀[결과] == 2이면 {
                코드바이트쓰기(73).    (* 0x49 REX.W+B *)
                코드바이트쓰기(186).   (* 0xBA = B8+r10하위 *)
                코드리틀엔디안32쓰기(하위).
                코드리틀엔디안32쓰기(상위).
                (* r10 → [rbp-오프셋] *)
                정수 오프셋은 임시변수오프셋풀[결과].
                오프셋을 인코딩_mov_mem_r10다.
            }
            아니면 {
                정수 오프셋은 임시변수오프셋풀[결과].
                코드바이트쓰기(73).
                코드바이트쓰기(186).
                코드리틀엔디안32쓰기(하위).
                코드리틀엔디안32쓰기(상위).
                오프셋을 인코딩_mov_mem_r10다.
            }
        }
        아니면 {
            (* 기존 imm32 경로 — 부호확장으로 충분한 경우 *)
            임시변수종류풀[결과] == 2이면 {
                코드바이트쓰기(73).
                코드바이트쓰기(199).
                코드바이트쓰기(194).
                코드리틀엔디안32쓰기(하위).
            }
            아니면 {
                정수 오프셋은 임시변수오프셋풀[결과].
                오프셋을 하위를 인코딩_mov_mem_imm32다.
            }
        }
    }
```


### Task 4: disp8 범위 조건 수정 (10곳)

**Files:**

**원인:** 모든 메모리 인코딩 함수에서 `(오프셋 >= -127) * (오프셋 < 0)`을 사용. 두 가지 문제:

2. `오프셋 < 0`이 양수 오프셋(0~+127)을 배제하여 항상 disp32로 인코딩

**수정 대상 행:** 350, 381, 402, 732, 751, 832, 849, 866, 884, 1072

- [ ] **Step 2: 일괄 수정**

모든 10곳에서:

**변경 전:**
```글
(오프셋 >= -127) * (오프셋 < 0) > 0이면 {
```

**변경 후:**
```글
(오프셋 >= -128) * (오프셋 <= 127) > 0이면 {
```

> **주의:** 양수 오프셋도 disp8으로 인코딩되도록 함. 이 경우 `변위 = 0 - 오프셋` 계산이 양수일 때도 올바른지 확인 필요. 양수 오프셋은 `[rbp + N]` 형태이므로 변위가 음수가 됨 → disp8 바이트는 2의 보수로 올바르게 인코딩됨.

- [ ] **Step 3: rbp+0 특수 케이스 처리 확인**

오프셋이 정확히 0인 경우를 확인. x86-64에서 `[rbp]`는 `mod=00, rm=101`이 RIP-relative로 예약되어 있으므로 반드시 `mod=01, disp8=0`으로 인코딩해야 함. 현재 스택 프레임 할당이 오프셋 0을 생성하지 않는지 확인하고, 방어 코드 추가:

```글
(오프셋 >= -128) * (오프셋 <= 127) > 0이면 {
    (* mod=01 disp8 경로 — 오프셋 0도 여기서 처리됨 *)
```

### Task 7: 레이블 검색 해시 테이블 적용

**Files:**
- Modify: `자체호스팅/PE생성기.글:100-140` (레이블검색, 패치모두해결)

**원인:** `레이블검색`이 선형 검색(O(N))으로, `패치모두해결`에서 N개 패치 × N개 레이블 = O(N²) 문자열 비교 발생. 대형 프로그램에서 성능 병목.

- [ ] **Step 1: 해시 테이블 선언 추가**

PE생성기.글 상단(레이블 관련 변수 선언 부근)에 해시 테이블 추가:

```글
정수 레이블해시크기 = 131072.
정수[131072] 레이블해시테이블.

[레이블해시초기화]는 {
    정수 번호 = 0.
    (번호 < 레이블해시크기)동안 {
        레이블해시테이블[번호] = -1.
        번호를 증가하다.
    }
}

[문자열 이름의 정수 길이의 레이블해시계산]은 → 정수 {
    정수 해시 = 5381.
    정수 번호 = 0.
    (번호 < 길이)동안 {
        해시 = 해시 * 33 + 이름[번호].
        번호를 증가하다.
    }
    정수 결과 = 해시 % 레이블해시크기.
    결과 < 0이면 { 결과 = 0 - 결과. }
    반환 결과.
}
```

- [ ] **Step 2: 레이블등록 시 해시 삽입 추가**

기존 `레이블등록` 함수에서 레이블 추가 후 해시 테이블에도 삽입:

```글
(* 기존 레이블 등록 후 *)
정수 슬롯 = 이름의 글자수(이름)의 레이블해시계산다.
정수 시도 = 0.
(시도 < 레이블해시크기)동안 {
    레이블해시테이블[슬롯] == -1이면 {
        레이블해시테이블[슬롯] = 레이블총수 - 1.
        탈출.
    }
    슬롯 = (슬롯 + 1) % 레이블해시크기.
    시도를 증가하다.
}
```

- [ ] **Step 3: 레이블검색을 해시 기반으로 교체**

```글
[문자열 이름으로 레이블검색]은 → 정수 {
    정수 슬롯 = 이름의 글자수(이름)의 레이블해시계산다.
    정수 시도 = 0.
    (시도 < 레이블해시크기)동안 {
        정수 인덱스는 레이블해시테이블[슬롯].
        인덱스 == -1이면 { 반환 -1. }
        문자열_비교(이름, 레이블이름풀[인덱스]) == 0이면 {
            반환 레이블오프셋풀[인덱스].
        }
        슬롯 = (슬롯 + 1) % 레이블해시크기.
        시도를 증가하다.
    }
    반환 -1.
}
```

- [ ] **Step 4: 초기화 호출 추가**

PE 생성 진입점 함수에서 `레이블해시초기화()`를 호출.
