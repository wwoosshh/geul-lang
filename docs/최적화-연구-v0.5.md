## 2. 즉시 적용 가능한 로우레벨 최적화

### 2.1 disp8 활성화 (코드 크기 ~15% 감소)

**현재 버그:** 모든 `인코딩_mov_mem_reg` 등에서 disp8 조건이 **항상 거짓**:

```글
(오프셋 <= -1) * (오프셋 > 0) > 0이면 {  (* 항상 false! *)
```

오프셋이 동시에 음수이고 양수일 수 없으므로, **항상 disp32 (4바이트)**를 사용합니다. 로컬 변수 대부분은 rbp-128 이내이므로 disp8 (1바이트)으로 충분합니다.

**수정:**
```글
(오프셋 >= -128) * (오프셋 <= 127) > 0이면 {
    (* disp8 사용 *)
}
```

**효과:** 함수당 수십~수백 바이트 코드 절약. 명령 캐시 효율 향상. fib(40)에서 ~5-10% 속도 개선 예상.

### 2.2 callee-saved push 조건부 생성

**현재:** 모든 함수에서 rbx, r12~r15를 무조건 push/pop (40바이트 + 10사이클).

**수정:** 레지스터 핀이 비활성인 함수에서는 push/pop 생략. 5번째+ 매개변수 오프셋도 동적으로 계산.

**효과:** 프롤로그/에필로그 40바이트 제거. fib(40) 같은 재귀 벤치에서 ~10-15% 개선.

### 2.3 레지스터 핀 재활성화

**현재 구현:** 함수 내 임시변수 사용 빈도를 분석하여 가장 자주 사용되는 변수를 rbx, r12~r15에 고정. 최대 5개 변수.

**비활성 이유:** 자체호스팅 과정에서 안정성 문제.

**재활성화 방법:**
1. 핀 수를 1개부터 시작하여 점진적으로 늘림
2. 매 단계에서 고정점 검증 (stage2 == stage3)
3. 안정성 확인 후 핀 수 증가

**효과:** 루프 카운터·인덱스가 레지스터에 상주하면 매 반복당 load/store 2회 제거. fib(40)에서 ~30-40% 개선 예상.

## 3. 중기 최적화 — 기존 패스 재활성화

### 3.1 피프홀 최적화

연속 명령 패턴 최적화:
- `mov [rbp-X], r10; mov r10, [rbp-X]` → 두 번째 load 제거
- `mov r10, [rbp-X]; mov [rbp-Y], r10; mov r10, [rbp-Y]` → 마지막 load 제거

### 3.2 죽은 저장 제거 (DSE)

`mov [rbp-X], r10` 이후 같은 슬롯을 다시 쓰면 첫 store 제거.

### 3.3 인라이닝

4개 이하 IR 명령의 소형 함수를 호출 위치에 직접 삽입. 함수 호출 오버헤드(call+프롤로그+에필로그+ret = ~20사이클) 제거.

### 3.4 LICM (루프 불변 코드 이동)

```글
(번호 < 글자수(이름))동안 {     (* 글자수(이름)은 매 반복 호출 *)
    ...
}
```
→ `글자수(이름)`을 루프 밖으로 이동.

## 4. 한국어 문법에서 오는 고유한 최적화 기회

### 4.1 조사 기반 인자 역할 추론

영어 문법에서는 인자의 역할이 **위치**로만 결정됩니다:
```c
copy(dest, src, size);  // 순서를 알아야 의미를 알 수 있음
```

글에서는 **조사**가 역할을 명시합니다:
```글
원본에서 대상으로 크기만큼 복사하다.
```

이 의미 정보를 IR에 보존하면:
- **별칭 분석(alias analysis):** `에서`(출처)는 읽기 전용, `으로`(목표)는 쓰기 → 자동으로 read/write 구분
- **병렬화:** 독립적인 인자(다른 조사)는 순서 무관하게 평가 가능
- **의도 기반 최적화:** `에` 조사는 대상이 변경됨을 암시 → 불필요한 복사 제거

**현재:** 조사 정보는 파서에서 인자 순서 번호로만 변환되어 버려짐.

### 4.2 SOV 어순의 컴파일 장점

**모든 인자가 동사(함수) 전에 결정됨:**
```글
5를 3에 10으로 함수하다.
(* 파서 시점: 인자 3개가 이미 완전히 파싱됨 → 함수 호출 *)
```

이것이 가능하게 하는 최적화:
- **인자 레지스터 사전 배정:** 인자 수를 미리 알므로, 임시 버퍼 없이 바로 RCX/RDX/R8/R9에 배치 가능
- **스필(spill) 최소화:** 인자 개수에 따라 필요한 레지스터를 미리 계획
- **인라이닝 결정:** 호출 전에 모든 인자 표현식의 복잡도를 알 수 있음

**현재:** 인자를 `PE_인자목록` 버퍼에 수집 후 방출하는 2단계 방식.

### 4.3 어미 기반 제어 흐름 패턴 인식

한국어 어미가 제어 흐름을 결정하므로, 어미 패턴으로 최적화 기회를 식별할 수 있습니다:

| 어미 패턴 | 의미 | 최적화 기회 |
|-----------|------|-------------|
| `~하고 ~하고 ~하다` | 연쇄 호출 | 중간 결과 레지스터 유지 |
| `~이면 { ~하다. }` | 단순 조건 | 분기 없는 cmov 변환 |
| `~동안 { ~하다. }` | 루프 | 루프 변수 핀, 언롤링 |
| `~를 반환하다` | SOV 반환 | 결과를 rax에 직접 생성 |

### 4.4 형태소 분석 단계 최적화

**접사 탐색 병목:** 32개 접사 × 37개 제외사전 = 단어당 최대 1,184회 비교

**개선안:**
- 접사 테이블을 **첫 바이트 기준 버킷**으로 분류 (한글 초성 14개)
- 제외사전을 **해시 테이블**로 변환 (현재 선형 탐색)
- 자주 사용되는 접사(`을/를/이/가/은/는`)를 우선 검사

## 5. 장기 최적화 — 새로운 패스

### 5.1 기본 레지스터 할당기

현재: 모든 임시변수 → 스택 슬롯. r10/r11만 캐시.

**선형 스캔 레지스터 할당:**
- 임시변수의 생존 범위(live range) 계산
- 겹치지 않는 변수를 같은 레지스터에 배정
- 사용 가능: rax, rcx, rdx, r8, r9 (caller-saved) + rbx, r12~r15 (callee-saved)
- 최대 9개 변수를 레지스터에 상주

**예상 효과:** C /Od 수준 (~1x) 도달 가능. fib(40) 770ms → ~350ms.

### 5.2 상수 전파 (Constant Propagation)

```글
정수 크기 = 10.
반복 (정수 번호 = 0: 번호 < 크기: 번호++) { }
```
→ `크기`가 상수이므로 `번호 < 10`으로 변환. 루프 경계가 즉시값이 되어 `cmp r10, 10` 단일 명령으로.

### 5.3 강도 감소 (Strength Reduction)

```글
정수 결과 = 번호 * 8.   → shl r10, 3
정수 결과 = 번호 * 10.  → lea r10, [r10 + r10*4]; shl r10, 1
```

### 5.4 루프 언롤링

작은 고정 횟수 루프를 펼쳐 분기 오버헤드 제거.

## 6. 남은 의존성 분석

### 6.1 msvcrt.dll 의존 (C 런타임)

생성된 바이너리는 **msvcrt.dll**에 완전히 의존합니다:

| 범주 | 함수 수 | 대체 가능성 |
|------|---------|-------------|
| 입출력 (printf 등) | 13개 | kernel32 WriteFile로 직접 구현 가능 |
| 메모리 (malloc 등) | 6개 | kernel32 HeapAlloc으로 대체 가능 |
| 문자열 (strcmp 등) | 10개 | 직접 구현 가능 (어셈블리) |
| 파일 (fopen 등) | 8개 | kernel32 CreateFile로 대체 가능 |
| 변환 (atoi 등) | 6개 | 직접 구현 가능 |
| 기타 (system 등) | 18개 | 부분적 대체 가능 |
| **합계** | **61개** | |

**완전 독립 경로:** kernel32.dll만 사용하여 모든 CRT 함수를 직접 구현하면 msvcrt.dll 의존 제거 가능. 하지만 작업량이 상당함.

### 6.2 kernel32.dll 의존

**제거 불가능:** Windows에서 실행하는 한 kernel32.dll은 필수. 이것은 의존성이 아닌 **플랫폼 API**입니다.

### 6.3 하드코딩된 경로

`네이티브컴파일러.글` 291행에 MSVC 절대 경로가 하드코딩:
```
C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Tools\MSVC\14.44.35207\...
```
이것은 `--emit-asm` 모드에서만 사용되므로 PE 직접 생성 경로에는 영향 없음.

### 6.4 std.gl 의존

`std.gl`은 C 함수의 한글 래퍼(`정의 쓰기 printf.` 등). 이것 자체는 단순 매크로이므로 의존성이 아닌 **편의 인터페이스**입니다.

### 6.5 의존성 요약

```
┌─────────────────────────────────┐
│  글 네이티브 컴파일러            │ ← 자체호스팅 (의존 없음)
│  (네이티브컴파일러.exe)          │
└─────────┬───────────────────────┘
          │ 생성
          ▼
┌─────────────────────────────────┐
│  사용자 프로그램 (.exe)          │
│  ├── kernel32.dll (필수, OS API) │
│  └── msvcrt.dll (C 런타임)  ◄── │ ← 유일한 제거 가능 의존성
└─────────────────────────────────┘
```

**컴파일러 자체**는 완전히 자체호스팅되어 외부 도구 의존이 없습니다.
**생성된 바이너리**는 msvcrt.dll에 의존하며, 이것이 유일한 제거 대상입니다.

## 7. 최적화 로드맵 (권장 순서)

### Phase 1: 즉시 (코드 변경 최소)
1. **disp8 버그 수정** — 조건문 논리 오류 수정만으로 ~10% 개선

7. **강도 감소**
