# 21. 리눅스 ELF 바이너리 생성 설계서

## 1. 목표

현재 네이티브 백엔드는 Windows PE 실행파일만 생성한다. 이 문서는 동일한 x86-64 기계어 인코더를 재사용하여 Linux ELF 실행파일을 생성하는 방법을 설계한다.

```
현재:   IR → x86-64 기계어 → PE 빌더 → .exe (Windows)
목표:   IR → x86-64 기계어 → PE 빌더 → .exe (Windows)
                            → ELF 빌더 → executable (Linux)
```

x86-64 명령어 인코더(`PE생성기.글`의 섹션 1~14)는 플랫폼 독립적이다. 바이트 버퍼, 레이블/패치 시스템, 모든 `인코딩_*` 함수를 그대로 재사용한다. 변경이 필요한 부분은 바이너리 포맷(PE 헤더 → ELF 헤더)과 임포트 메커니즘(IAT → PLT/GOT 또는 syscall)뿐이다.

---

## 2. 아키텍처

### 2.1 현재 PE생성기 구조 (4,295줄)

```
PE생성기.글 구조:
  섹션 1:   바이트 버퍼 (코드버퍼, 데이터버퍼)          ← 재사용
  섹션 2:   레이블 + 패치 시스템                        ← 재사용
  섹션 3:   함수 맹글링 (한글 → ASCII)                  ← 재사용
  섹션 4:   함수별 코드 오프셋                          ← 재사용
  섹션 5:   x86-64 명령어 인코더 (~60개 함수)           ← 재사용
  섹션 6-14: IR → 기계어 변환 루프                      ← 재사용
  섹션 15:  정렬 올림 유틸리티                          ← 재사용
  섹션 16:  PE 파일 쓰기 (임포트, 헤더, 출력)           ← ELF 대체 필요
```

약 3,600줄(섹션 1~15)은 그대로 재사용하고, 약 700줄(섹션 16)을 ELF 버전으로 대체/추가한다.

### 2.2 새 파일 구조

```
자체호스팅/
├── PE생성기.글              ← 기존 유지 (섹션 1~15 공용 + 섹션 16 PE 전용)
├── ELF생성기.글             ← 신규: ELF 헤더 + 동적 링킹 또는 syscall
├── 네이티브컴파일러.글       ← 수정: -대상 옵션으로 PE/ELF 선택
```

대안으로, PE생성기.글 안에 `만약정의` 분기를 사용하여 단일 파일로 유지할 수도 있다. 그러나 PE와 ELF의 차이가 크므로 별도 파일이 깔끔하다.

## 3. PE와 ELF 비교

### 3.1 핵심 차이

| 항목 | PE (Windows) | ELF (Linux) |
|------|-------------|-------------|
| 매직 바이트 | `"MZ"` (0x4D5A) + `"PE\0\0"` (0x50450000) | `"\x7fELF"` (0x7F454C46) |
| 헤더 크기 | DOS(64) + PE시그니처(4) + COFF(20) + Optional(240) + 섹션헤더(40*N) = ~512바이트 | ELF헤더(64) + 프로그램헤더(56*N) = ~232바이트 |
| 이미지 베이스 | 0x00400000 (고정, 재배치 가능) | 0x00400000 (정적) 또는 PIE |
| 진입점 | ImageBase 기준 상대 주소 (RVA) | 절대 가상 주소 |
| 코드 섹션 | `.text` (섹션 헤더로 기술) | `.text` (프로그램 헤더 PT_LOAD) |
| 읽기전용 데이터 | `.rdata` | `.rodata` |
| 초기화 안 된 데이터 | `.bss` (섹션 헤더) | `.bss` (PT_LOAD, MEMSZ > FILESZ) |
| 외부 함수 호출 | Import Directory + IAT (DLL 이름 + 함수 이름) | `.dynamic` + `.dynsym` + PLT/GOT |
| 프로세스 종료 | `call ExitProcess` (kernel32.dll) | `mov rax, 60; syscall` |
| 출력 (printf 대안) | `call printf` (ucrtbase.dll) | `mov rax, 1; syscall` (write) |
| 파일 정렬 | 512바이트 (FileAlignment) | 없음 (또는 페이지 4096) |
| 섹션 정렬 | 4096바이트 (SectionAlignment) | 4096바이트 (페이지 크기) |

### 3.2 현재 PE 생성기의 임포트 함수 목록

PE생성기.글에서 임포트하는 함수:

**kernel32.dll (5개):**
- ExitProcess, QueryPerformanceCounter, QueryPerformanceFrequency, GetStdHandle, WriteConsoleA

**ucrtbase.dll (최대 64개, 실제 사용 ~40개):**
- printf, malloc, free, exit, strcmp, strlen, fopen, fclose, fread, fwrite, ...

ELF에서는 이 함수들을 다음 두 방법 중 하나로 대체한다:
1. **libc.so.6 동적 링킹** — PLT/GOT를 통해 동일 함수 호출 (printf, malloc 등)
2. **Linux syscall 직접** — libc 없이 커널 인터페이스만 사용 (제한적)

## 4. ELF 파일 구조 (상세)

### 4.1 최소 정적 실행파일 (1단계: syscall만)

```
오프셋    크기    내용
─────────────────────────────────────────
0x0000    64      ELF 헤더
0x0040    56      프로그램 헤더 #0 (PT_LOAD: .text)
0x0078    56      프로그램 헤더 #1 (PT_LOAD: .rodata + .bss)
0x00B0    --      패딩 (페이지 정렬까지)
0x1000    N       .text 섹션 (기계어 코드)
0x1000+N  --      패딩
0x2000    M       .rodata 섹션 (문자열 상수)
0x2000+M  --      .bss (파일에 데이터 없음, 메모리만 확보)
─────────────────────────────────────────
총 크기: ~8KB + 코드 + 데이터
```

### 4.2 ELF 헤더 (64바이트)

```
오프셋  크기  필드                    값                설명
──────────────────────────────────────────────────────────────
0x00    4    e_ident[EI_MAG]         7F 45 4C 46       "\x7fELF"
0x04    1    e_ident[EI_CLASS]       02                64비트
0x05    1    e_ident[EI_DATA]        01                리틀엔디안
0x06    1    e_ident[EI_VERSION]     01                현재 버전
0x07    1    e_ident[EI_OSABI]       00                ELFOSABI_NONE (리눅스)
0x08    8    e_ident[패딩]           00 00 00 00 ...   0으로 채움
0x10    2    e_type                  0002              ET_EXEC (실행파일)
0x12    2    e_machine               003E              EM_X86_64
0x14    4    e_version               00000001          현재 버전
0x18    8    e_entry                 0000000000401000   진입점 (0x400000 + 0x1000)
0x20    8    e_phoff                 0000000000000040   프로그램 헤더 오프셋 (64)
0x28    8    e_shoff                 0000000000000000   섹션 헤더 오프셋 (0 = 없음)
0x30    4    e_flags                 00000000          플래그 없음
0x34    2    e_ehsize                0040              ELF 헤더 크기 (64)
0x36    2    e_phentsize             0038              프로그램 헤더 항목 크기 (56)
0x38    2    e_phnum                 0002              프로그램 헤더 개수 (2)
0x3A    2    e_shentsize             0040              섹션 헤더 항목 크기 (64)
0x3C    2    e_shnum                 0000              섹션 헤더 개수 (0)
0x3E    2    e_shstrndx              0000              섹션 이름 문자열 인덱스 (0)
```

### 4.3 프로그램 헤더 (각 56바이트)

**프로그램 헤더 #0: .text (코드)**

```
오프셋  크기  필드          값                        설명
────────────────────────────────────────────────────────────
0x00    4    p_type        00000001                  PT_LOAD
0x04    4    p_flags       00000005                  PF_R | PF_X (읽기+실행)
0x08    8    p_offset      0000000000001000          파일 내 오프셋
0x10    8    p_vaddr       0000000000401000          가상 주소
0x18    8    p_paddr       0000000000401000          물리 주소 (= vaddr)
0x20    8    p_filesz      코드버퍼위치               파일 내 크기
0x28    8    p_memsz       코드버퍼위치               메모리 내 크기
0x30    8    p_align       0000000000001000          정렬 (4096)
```

**프로그램 헤더 #1: .rodata + .bss (데이터)**

```
오프셋  크기  필드          값                        설명
────────────────────────────────────────────────────────────
0x00    4    p_type        00000001                  PT_LOAD
0x04    4    p_flags       00000006                  PF_R | PF_W (읽기+쓰기)
0x08    8    p_offset      데이터파일오프셋            파일 내 오프셋
0x10    8    p_vaddr       0000000000402000          가상 주소
0x18    8    p_paddr       0000000000402000          물리 주소
0x20    8    p_filesz      데이터버퍼위치              파일 내 크기 (.rodata만)
0x28    8    p_memsz       데이터+BSS크기             메모리 내 크기 (.rodata + .bss)
0x30    8    p_align       0000000000001000          정렬 (4096)
```

> **핵심**: BSS는 `p_memsz > p_filesz`로 표현한다. 파일에 데이터를 쓰지 않고도 메모리를 확보할 수 있다. PE에서는 별도 섹션 헤더가 필요했지만, ELF에서는 프로그램 헤더 하나로 처리한다.

## 5. 시스템 호출 (PE 임포트 대체)

### 5.1 Linux x86-64 syscall 규약

```
syscall 번호: rax
인자 순서:    rdi, rsi, rdx, r10, r8, r9
반환값:       rax
```

**주의**: Windows x64 호출 규약(rcx, rdx, r8, r9)과 다르다. PE생성기의 IR→기계어 변환에서 인자 전달 레지스터를 변경해야 한다.

### 5.2 PE 임포트 → Linux syscall 매핑

| PE 함수 | Linux syscall | 번호(rax) | 인자 |
|---------|---------------|----------|------|
| ExitProcess(code) | exit(code) | 60 | rdi=code |
| WriteConsoleA(h,buf,len,...) | write(fd,buf,len) | 1 | rdi=fd, rsi=buf, rdx=len |
| — | read(fd,buf,len) | 0 | rdi=fd, rsi=buf, rdx=len |

### 5.3 syscall 인코딩

```
write(1, 문자열주소, 길이):
    mov rax, 1          → 48 C7 C0 01 00 00 00
    mov rdi, 1          → 48 C7 C7 01 00 00 00
    lea rsi, [rip+N]    → 48 8D 35 [rel32]
    mov rdx, 길이        → 48 C7 C2 [imm32]
    syscall              → 0F 05

exit(code):
    mov rax, 60         → 48 C7 C0 3C 00 00 00
    mov rdi, code       → 48 C7 C7 [imm32]
    syscall              → 0F 05
```

`syscall` 명령어 인코딩은 `0F 05` (2바이트)이다. PE생성기에 `인코딩_syscall` 함수를 추가한다:

```글
[인코딩_syscall]은 {
    15를 코드바이트쓰기다.     (* 0x0F *)
    5를 코드바이트쓰기다.      (* 0x05 *)
}
```

## 6. 구현 단계

### 6.1 1단계: 최소 ELF (syscall만, libc 없음)

printf/malloc 없이, write syscall과 exit syscall만으로 동작하는 최소 실행파일을 생성한다.

**목표 프로그램:**
```글
[시작하기]는 → 정수 {
    (* write(1, "Hello\n", 6) *)
    (* exit(0) *)
    반환 0.
}
```

**생성되는 바이너리:**
```
ELF 헤더 (64바이트)
프로그램 헤더 x2 (112바이트)
패딩 (0x1000까지)
.text: 기계어 (~50바이트)
패딩 (0x2000까지)
.rodata: "Hello\n" (6바이트)
```

**구현할 함수:**

```글
(* ELF생성기.글 *)

(* --- ELF 헤더 버퍼 --- *)
문자[4096] ELF_헤더.
정수 ELF_헤더위치 = 0.

[정수 바이트를 ELF_헤더바이트]는 { ... }    (* PE_헤더바이트와 동일 패턴 *)
[정수 값을 ELF_헤더16비트]는 { ... }
[정수 값을 ELF_헤더32비트]는 { ... }
[정수 값을 ELF_헤더64비트]는 { ... }        (* PE는 하위32+0, ELF는 진짜 64비트 *)

[ELF_헤더구성]은 { ... }
[ELF_프로그램헤더구성]은 { ... }
[문자열 경로를 ELF_파일출력]은 { ... }
[문자열 경로를 ELF_파일쓰기]는 { ... }      (* PE_파일쓰기에 대응 *)
```

**PE 함수와의 대응:**

| PE생성기 함수 | ELF생성기 대응 | 비고 |
|--------------|---------------|------|
| `PE_헤더바이트` | `ELF_헤더바이트` | 동일 로직 |
| `PE_헤더16비트` | `ELF_헤더16비트` | 동일 로직 |
| `PE_헤더32비트` | `ELF_헤더32비트` | 동일 로직 |
| `PE_헤더64비트` | `ELF_헤더64비트` | **변경**: PE는 32+0, ELF는 진짜 64비트 필요 |
| `PE_헤더구성` (131줄) | `ELF_헤더구성` (~30줄) | ELF 헤더가 훨씬 단순 |
| `PE_임포트테이블구성` (~200줄) | 불필요 (1단계) | syscall은 임포트 불필요 |
| `PE_썽크코드생성` (~80줄) | 불필요 (1단계) | — |
| `PE_파일출력` (36줄) | `ELF_파일출력` (~25줄) | 구조 단순 |
| `PE_파일쓰기` (41줄) | `ELF_파일쓰기` (~30줄) | 단계 축소 |

**예상 줄수: ~200줄**

### 6.2 2단계: libc 동적 링킹 (printf, malloc 사용)

1단계에서 ELF 형식이 검증되면, libc.so.6 동적 링킹을 추가한다.

**추가 구조:**

```
+---------------------------+
| ELF Header (64 bytes)     |
| Program Header: PT_LOAD   |  ← .text
| Program Header: PT_LOAD   |  ← .rodata + .bss
| Program Header: PT_DYNAMIC|  ← 동적 링킹 정보
| Program Header: PT_INTERP |  ← "/lib64/ld-linux-x86-64.so.2"
+---------------------------+
| .interp                   |  ← 동적 링커 경로
| .text                     |  ← 기계어 + PLT 스텁
| .rodata                   |  ← 문자열 상수
| .dynamic                  |  ← DT_NEEDED "libc.so.6", DT_SYMTAB, etc.
| .dynsym                   |  ← 심볼 테이블 (printf, malloc, ...)
| .dynstr                   |  ← 문자열 테이블 ("printf\0", "malloc\0", ...)
| .rela.plt                 |  ← PLT 재배치 엔트리
| .got.plt                  |  ← GOT (동적 링커가 런타임에 주소 기록)
+---------------------------+
```

**PLT/GOT 메커니즘:**

```
사용자 코드:  call printf_PLT
                    ↓
PLT 스텁:    jmp [GOT_printf]      ← GOT에 저장된 주소로 점프
             push 재배치_인덱스
             jmp PLT[0]            ← 동적 링커 호출 (최초 1회)
                    ↓
GOT 엔트리:  [처음에는 PLT push 명령 주소]
             [동적 링커가 실제 printf 주소로 패치]
```

PLT 스텁은 함수당 16바이트:

```
ff 25 [GOT오프셋]    jmp QWORD PTR [rip+GOT오프셋]   (6바이트)
68 [인덱스]           push 재배치인덱스                 (5바이트)
e9 [PLT[0]오프셋]    jmp PLT[0]                       (5바이트)
```

**PE IAT와의 비교:**

| 항목 | PE (현재 구현) | ELF PLT/GOT |
|------|---------------|-------------|
| 간접 호출 | `call [rip+IAT오프셋]` (FF 15) | `call PLT_stub` (E8) → `jmp [GOT]` |
| 테이블 위치 | .rdata 섹션 내 | .got.plt 섹션 (별도) |
| 해결 시점 | 로더가 프로세스 시작 시 | 지연 바인딩 (최초 호출 시) |
| 썽크 코드 | PE생성기의 `PE_썽크코드생성` | PLT 스텁 (유사한 역할) |
| DLL/SO 이름 | Import Directory에 문자열 | `.dynamic` DT_NEEDED |

현재 PE생성기의 썽크 코드(`ff 25 [rip+...]`)는 ELF의 GOT 간접 점프와 거의 동일하므로, 이 패턴을 재사용할 수 있다.

**예상 추가 줄수: ~400줄** (동적 섹션, PLT/GOT, .interp, 심볼 테이블)

### 6.3 3단계: 호출 규약 전환

Linux에서 libc 함수를 호출하려면 System V AMD64 ABI를 따라야 한다:

| 인자 번호 | Windows x64 | System V AMD64 (Linux) |
|----------|-------------|----------------------|
| 1번째 | rcx | rdi |
| 2번째 | rdx | rsi |
| 3번째 | r8 | rdx |
| 4번째 | r9 | rcx |
| 5번째+ | 스택 | r8, r9, 스택 |
| 반환값 | rax | rax |

**PE생성기 변경 범위:**

현재 PE생성기의 `명령_인자전달` 처리:

```
인자0 → mov rcx, 값     (Windows)
인자1 → mov rdx, 값     (Windows)
인자2 → mov r8, 값      (Windows)
인자3 → mov r9, 값      (Windows)
```

Linux 대상일 때:

```
인자0 → mov rdi, 값     (Linux)
인자1 → mov rsi, 값     (Linux)
인자2 → mov rdx, 값     (Linux)
인자3 → mov rcx, 값     (Linux)
인자4 → mov r8, 값      (Linux)
인자5 → mov r9, 값      (Linux)
```

IR→기계어 변환 루프에서 대상 플랫폼 전역 변수(`ELF_대상여부`)를 확인하여 레지스터를 분기한다. 변경 범위는 `명령_인자전달` 처리 부분 약 20줄이다.

## 8. 64비트 값 처리 주의점

현재 PE생성기의 `PE_헤더64비트` 함수:

```글
[정수 값을 PE_헤더64비트]는 {
    PE_헤더32비트(값).
    PE_헤더32비트(0).       (* 상위 32비트를 항상 0으로 *)
}
```

이는 PE에서 ImageBase(0x00400000)가 32비트 범위이므로 작동한다. 그러나 ELF에서도 동일하게 0x400000 범위를 사용하므로 같은 방식으로 충분하다.

만약 향후 PIE(Position Independent Executable)나 높은 주소를 사용할 경우, 진짜 64비트 쓰기가 필요하다:

```글
[정수 하위를 정수 상위를 ELF_헤더진짜64비트]는 {
    ELF_헤더32비트(하위).
    ELF_헤더32비트(상위).
}
```

1단계에서는 불필요하다.

## 10. 네이티브컴파일러 통합

### 10.1 명령줄 옵션

```
네이티브컴파일러 소스.글                      → PE (기본, Windows)
네이티브컴파일러 소스.글 -대상 윈도우          → PE
네이티브컴파일러 소스.글 -대상 리눅스          → ELF
네이티브컴파일러 소스.글 -대상 리눅스 -출력 a  → ELF, 파일명 "a"
```

### 10.2 코드 분기

```글
(* 네이티브컴파일러.글 *)
정수 대상_리눅스 = 0.     (* 0=Windows, 1=Linux *)

(* 명령줄 처리에서 *)
문자열_비교(인자, "-대상") == 0이면 {
    문자열_비교(다음인자, "리눅스") == 0이면 {
        대상_리눅스 = 1.
    }
}

(* 코드 생성 후 *)
대상_리눅스 == 1이면 {
    출력경로를 ELF_파일쓰기다.
}
아니면 {
    출력경로를 PE_파일쓰기다.
}
```

## 11. 검증 계획

### 11.1 1단계 검증

1. Windows에서 ELF 바이너리 생성 (`네이티브컴파일러 test.글 -대상 리눅스`)
