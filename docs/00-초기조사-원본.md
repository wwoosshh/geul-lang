# Research: Korean (Hangul) C-like Programming Language

## 1. Existing Korean Programming Languages

### Aheui (아희) - Esoteric
- **Type**: Esoteric / 2D grid-based language (similar to Befunge)
- **Created**: 2005 by JongYoon Lee
- **How it works**: Programs are written as a 2D grid of Hangul characters. Consonants specify instructions (arithmetic, stack manipulation, I/O), vowels control cursor movement direction. A cursor traverses the grid with position and momentum.
- **Name**: Archaic Korean for "child/youngster"
- **Implementations**: Python, Go, Julia, and others
- **Limitation**: Purely esoteric. Not designed for practical programming. No variables, functions, or standard control flow. Interesting as art/culture but not usable for real software.
- **GitHub**: https://github.com/aheui

### Ssiat (씨앗 / Seed) - Educational/Practical
- **Type**: Korean-syntax language developed by Nanum Technology (1994)
- **Key features**:
  - Grammatical structure mirrors Korean sentence structure
  - Statements end with periods (.) instead of semicolons
  - Numbers must be written in Korean (e.g., "이백이십" for 220)
  - Coding uses only Korean characters and spaces
  - Error messages in Korean
  - Similar structure to C but with Korean grammar
- **Limitation**: Proprietary, limited ecosystem, numbers-in-Korean requirement makes arithmetic cumbersome. Not actively maintained or widely adopted.

### Saesak (새싹) - Educational/Object-Oriented
- **Type**: Korean object-oriented programming language based on Java
- **Focus**: Programming education for Korean students
- **Key features**:
  - Grammatically correct Korean syntax - reads like natural Korean sentences
  - Supports OOP and lambda expressions
  - IDE with Korean error messages
  - 10% more concise than other Korean languages on average
  - 24% more concise than widely-used mainstream languages
- **Academic paper**: Published in KCI (Korea Citation Index)
- **Limitation**: Academic/educational focus, not widely adopted for production use

### Other Historical Korean Languages
- **KIST/KAIST (mid-1970s)**: First Korean programming attempt - a preprocessor that converted Korean commands to FORTRAN. Simple text substitution approach.
- **Hongik University (early 1980s)**: Korean language based on Pascal concepts
- **Malssee (말씨)**: Uses only Korean characters and spaces
- **Soom (숨)**: Korean programming language
- **Hanpul (한플)**: Korean programming language
- **Dolittle (두리틀)**: Korean version of educational language

### Summary of Korean Language Limitations
1. Most are esoteric (Aheui) or educational (새싹)
2. None achieved full C-level systems programming capability
3. Limited/no ecosystem (no package managers, no libraries)
4. Most are abandoned or unmaintained
5. No Korean language has successfully mapped to full C functionality
6. The preprocessor-to-FORTRAN approach from the 1970s is the closest precedent to a "transpiler to C" idea

## 2. Serious Non-English Programming Languages

### Wenyan-lang (文言) - Classical Chinese
- **GitHub**: https://github.com/wenyan-lang/wenyan (20,200+ stars, 1,100+ forks)
- **Creator**: Lingdong Huang (Carnegie Mellon undergrad, created in 4 days)
- **Transpiles to**: JavaScript, Python, Ruby, C++
- **Features**:
  - Syntax follows classical Chinese literary grammar
  - Online IDE with file explorer, auto-complete, dark mode
  - Can render code as traditional Chinese book pages (SVG)
  - Package manager called "wyg" (文淵閣)
  - VS Code extension with side-by-side compiled JS view
- **Significance**: Most successful non-English language project by GitHub stars. Demonstrates that a transpiler approach to a mainstream target (JS) can gain significant traction.
- **Limitation**: Still fundamentally esoteric/artistic. Not used for production software.

### Easy Programming Language / EPL (易语言) - Chinese
- **Type**: Visual compiled multilingual proprietary language
- **Significance**: Actually commercially successful in China
- **Features**:
  - Full Chinese development environment
  - Compiles to standalone executables
  - Supports: Windows Forms, Console Apps, DLLs, Linux Console
  - Rich library ecosystem: ADO Database, networking, multimedia, compression, etc.
  - RAD (Rapid Application Development) visual designer
  - Variants: Simplified Chinese, Traditional Chinese, English, Japanese
- **Status**: The most successful non-English programming language by actual usage. Proves that a full-featured Chinese-keyword language can achieve commercial viability.
- **Caveat**: Proprietary, Windows-focused, sometimes associated with malware development

### Qalb (قلب) - Arabic
- **Creator**: Ramsey Nasser (2012, Eyebeam Art + Technology Center)
- **Type**: Functional language with Lisp/Scheme-like syntax
- **Implementation**: Tree-walking interpreter in JavaScript (based on Peter Norvig's Lispy)
- **Features**: Right-to-left text layout, Lebanese Arabic keywords, parenthesized list syntax
- **Significance**: Well-known artistic/academic project highlighting Anglophone bias in programming
- **Limitation**: Minimal primitives, not production-ready, artistic project

### Qi (棋) - Chinese
- **GitHub**: https://github.com/AnonymousAAArdvark/qi
- **Type**: Lightweight programming language designed to be written in Chinese
- **Relevant as**: A modern attempt at Chinese-first language design

### Rapira - Russian
- **Type**: Interpreted procedural language with strong dynamic typing
- **Historical significance**: One of the earlier non-English programming languages from the Soviet era
- **Limitation**: Historical, not actively used

### Robik - Russian
- **Type**: Another Soviet-era programming language
- **Historical significance**: Part of the Soviet computing tradition of Russian-language programming

### Hedy - Multilingual Educational
- **Supports**: 47 languages including Dutch, Spanish, Arabic, Turkish, Chinese, Hindi
- **Key insight**: Uses localized keywords AND can change ordering of syntactic constructs per language
- **Technical approach**: "Rich formalism for defining modifications of context-free grammars that can be compiled into parsers" - handles not just keyword translation but grammar reordering
- **Significance**: Proves that a single language infrastructure can support multiple natural language syntaxes, including non-Latin scripts

### Chinese Python / ZhPy (周蟒)
- **Created**: 2007
- **Type**: Python module that converts Chinese keywords/variables/parameters to standard Python
- **Approach**: Source-to-source translation (Chinese Python -> standard Python)
- **Supports**: Simplified and Traditional Chinese
- **Significance**: Demonstrates the "thin translation layer" approach - minimal effort, maximum compatibility

## 3. Technical Approaches to Building a Korean C-like Language

### Approach A: Preprocessor Macro (#define) Approach
**Simplest, least effort, most limited**

```c
// hangul_c.h
#define 정수 int
#define 실수 float
#define 문자 char
#define 반환 return
#define 만약 if
#define 아니면 else
#define 동안 while
#define 반복 for
#define 구조체 struct
#define 포함 #include
#define 공허 void
#define 주함수 main
// ... etc
```

**Pros**:
- Trivial to implement (just a header file)
- Full C compatibility - it IS C
- Works with all existing C tools, debuggers, IDEs
- Zero runtime overhead

**Cons**:
- Cannot change syntax/grammar (stuck with C's English grammar order)
- Cannot handle Korean postpositions or verb endings
- Error messages still in English referring to C keywords
- Identifiers in Korean work already (C11/C23 support Unicode identifiers)
- Very superficial - just keyword aliasing
- Cannot change operators or introduce Korean-natural constructs
- The `#define` approach requires identifiers, so Korean keywords must be valid C identifiers (they are in C11+ with Unicode support)

### Approach B: Transpiler (Korean Source -> C Source -> Compile)
**Recommended approach - best balance of effort and capability**

Architecture:
```
Korean Source (.한 files)
    |
    v
[Lexer] -- tokenizes Korean + Unicode source
    |
    v
[Parser] -- builds AST using Korean grammar rules
    |
    v
[AST Transformation] -- intermediate representation
    |
    v
[C Code Generator] -- emits standard C code
    |
    v
Standard C Source (.c files)
    |
    v
[gcc/clang] -- compile to native binary
```

**Pros**:
- Can define truly Korean grammar (SOV order, postpositions, verb endings)
- Full C functionality via generated C code
- Leverage existing C compilers for optimization, platform support
- Can generate readable C for debugging
- Can add Korean-specific syntax sugar
- Historical precedent: KIST 1970s approach, Wenyan-lang, ZhPy all use this

**Cons**:
- Error mapping: C compiler errors reference generated C, not Korean source
  - Mitigation: Use `#line` directives to map back to Korean source
- Build step complexity
- Need to maintain transpiler as C standard evolves

**Key implementation detail**: Use `#line` preprocessor directives in generated C to map error locations back to the Korean source file.

### Approach C: Custom Compiler (Korean Source -> Assembly/Machine Code)
**Maximum control, maximum effort**

**Pros**:
- Full control over error messages (all in Korean)
- Can optimize for Korean-specific constructs
- No dependency on C toolchain for language semantics

**Cons**:
- Enormous implementation effort
- Must handle all target architectures
- Must implement all optimizations from scratch
- Not practical for a new project - even major languages (Rust, Zig, early C++) started as transpilers

### Approach D: Interpreter
**Good for prototyping, bad for C-like performance goals**

- Qalb uses this approach (tree-walking interpreter in JS)
- Good for educational/prototype purposes
- Cannot deliver C-like performance
- Useful as a first step before building a transpiler

### Recommended Strategy: Phased Approach
1. **Phase 1**: Preprocessor macros for keyword mapping (proof of concept)
2. **Phase 2**: Simple transpiler for basic Korean->C translation
3. **Phase 3**: Full transpiler with Korean grammar support, error mapping, standard library wrappers

## 4. Lexer/Parser Tool Options

### ANTLR4
- **Best Unicode support**: Full Unicode (U+0000 to U+10FFFF) since ANTLR 4.7
- **Korean character ranges**: Can specify Hangul Syllables block (U+AC00-U+D7AF) and Jamo blocks directly
- **Target languages**: Java, C#, Python, JavaScript, Go, C++, Swift, Dart, PHP
- **Grammar format**: EBNF-like, separate lexer and parser grammars
- **Pros**: Mature, well-documented, large grammar repository (grammars-v4), excellent Unicode support
- **Cons**: Runtime dependency, may be heavy for a simple transpiler

### RE/flex (Recommended for C/C++ target)
- **Type**: Modern Flex replacement for C++
- **Unicode**: Full UTF-8/16/32 support with auto-detection
- **Features**: Unicode property matching \p{}, seamless Bison integration
- **Korean support**: Supports Unicode identifier matching including CJK/Hangul ranges
- **Pros**: Drop-in Flex replacement, high performance, full Unicode, integrates with Bison
- **Cons**: C++ only

### Flex + Bison (Traditional)
- **Limitation**: Flex has poor Unicode support natively
- **Workaround**: Process UTF-8 bytes manually, or use a wrapper
- **Pros**: Extremely well-known, tons of documentation, C output
- **Cons**: Unicode handling is painful, not recommended for Korean

### Hand-written Lexer/Parser
- **Approach**: Write lexer and recursive descent parser manually in C
- **Pros**: Full control, no dependencies, can handle Korean-specific edge cases
- **Cons**: More initial work, but arguably simpler for a language that maps closely to C
- **Note**: Many production compilers use hand-written parsers (GCC, Clang, Go, Rust)

### Recommendation
- **For rapid prototyping**: ANTLR4 (best Unicode support, fast grammar iteration)
- **For production C-targeting transpiler**: RE/flex + Bison or hand-written lexer + recursive descent parser
- **Avoid**: Plain Flex for Korean/Unicode work

## 5. Key Technical Challenges

### 5.1 Unicode/UTF-8 Handling in Lexer

**Hangul Unicode Ranges**:
- Hangul Syllables: U+AC00 - U+D7AF (11,172 precomposed syllables)
- Hangul Jamo: U+1100 - U+11FF (conjoining jamo)
- Hangul Compatibility Jamo: U+3130 - U+318F
- Hangul Jamo Extended-A: U+A960 - U+A97F
- Hangul Jamo Extended-B: U+D7B0 - U+D7FF

**Critical Challenge: Normalization**
- The same Korean text can be represented as:
  - Precomposed syllables (NFC): Single code point per syllable (e.g., U+D55C for 한)
  - Decomposed jamo (NFD): Separate lead, vowel, tail jamo (e.g., U+1112 + U+1161 + U+11AB for 한)
- **Solution**: Normalize all input to NFC before lexing. Python uses NFKC for identifiers.
- macOS file systems use NFD, which means the same Korean filename/source can look different on macOS vs Windows/Linux
- **Must implement**: NFC normalization at lexer input stage

**Lexer Implementation**:
- Read UTF-8 byte sequences, decode to code points
- Classify code points: Is it Hangul? Is it ASCII? Is it whitespace?
- For identifiers: First character must be Hangul syllable or jamo; subsequent characters can be Hangul, ASCII digits, underscore
- Fast path for ASCII (single byte), slow path for multi-byte UTF-8

### 5.3 Korean Grammar Differences

**Key challenge**: Korean is SOV (Subject-Object-Verb) while C follows English SVO patterns.

**Option 1: Keep C syntax, just translate keywords** (Simplest)
```
정수 시작() {
    정수 가 = 10;
    만약 (가 > 5) {
        출력("크다\n");
    }
    반환 0;
}
```

**Option 2: Korean-natural grammar with postpositions** (Most ambitious)
Korean uses postpositions (조사) that indicate grammatical role:
- 을/를 (-eul/-reul): object marker
- 이/가 (-i/-ga): subject marker
- 에 (-e): location/destination
- 로/으로 (-ro/-euro): means/direction

Example of more natural Korean syntax:
```
시작 함수:
    정수 값 = 10 이다.
    만약 값이 5보다 크면:
        "크다"를 출력하라.
    반환 0.
```

**Recommendation**: Start with Option 1 (keyword translation preserving C syntax). This is what EPL (易语言), ZhPy, and the KIST FORTRAN preprocessor all did. Option 2 requires significantly more parser work and creates a language that diverges from C, making it harder for users to transition to standard C.

### 5.4 Standard Library Mapping

Two approaches:
1. **Wrapper headers**: Create Korean-named wrapper functions that call C standard library
2. **Transpiler rewriting**: Have the transpiler replace Korean function names with C equivalents

Example wrapper approach:
```c
// 표준라이브러리.h (standard library header)
#define 출력 printf
#define 입력 scanf
#define 할당(크기) malloc(크기)
#define 해제(포인터) free(포인터)
#define 문자열길이(문자열) strlen(문자열)
#define 문자열복사(목적지, 원본) strcpy(목적지, 원본)
```

### 5.5 IDE/Editor Support

**Challenges**:
- Syntax highlighting for Korean keywords
- Auto-completion with Korean input (IME interaction)
- Error messages mapping back to Korean source
- Debugger integration (stepping through Korean source, not generated C)

**Implementation options**:
- **VS Code extension**: Language Server Protocol (LSP) + TextMate grammar for syntax highlighting. Wenyan-lang has a successful VS Code extension as precedent.
- **Tree-sitter grammar**: For editors like Neovim, Helix. Tree-sitter handles Unicode well.
- **Source maps**: Similar to JavaScript source maps, map generated C positions back to Korean source positions. Use C `#line` directives.

## 6. Similar Successful Projects and Lessons

### Wenyan-lang (文言) - Key Lessons
- Transpiler to JS/Python/Ruby/C++ works well
- Beautiful presentation (rendering as traditional book pages) generated massive interest
- Package manager and IDE were crucial for adoption
- 20K+ GitHub stars shows cultural/linguistic programming languages can gain traction
- Created by one person in 4 days (initial version) - proves small-scope is viable

### EPL / 易语言 - Key Lessons
- The ONLY commercially successful non-English programming language
- Full IDE with visual designer was key to adoption
- Complete Chinese environment (menus, docs, errors, everything)
- Proves: if you go all-in on native language experience, people will use it
- Compilation to native code (not interpretation) was important for credibility

### ZhPy (Chinese Python) - Key Lessons
- Thin translation layer approach: Chinese -> Python source -> Python interpreter
- Minimal effort, maximum compatibility
- But also minimal adoption - "just keyword translation" wasn't compelling enough
- Lesson: Need to go beyond just keyword mapping to be interesting

### Hedy - Key Lessons
- Supports 47 languages with localized keywords AND grammar reordering
- Proves that a single parser infrastructure can handle multiple natural language syntaxes
- "Rich formalism for defining modifications of context-free grammars" - technically sophisticated
- Grammar modifications per language, not just keyword substitution
- Educational focus limits it to beginners, but the technical approach is applicable

### KIST Korean FORTRAN (1975) - Key Lessons
- The very first Korean programming language was a preprocessor/transpiler
- Simple text substitution from Korean to FORTRAN
- Proves the transpiler approach has historical precedent in Korean computing

## 7. Recommended Implementation Strategy

### Phase 1: Proof of Concept (1-2 weeks)
- Header file with `#define` macros mapping Korean keywords to C
- Test with simple programs
- Validates keyword choices and basic usability
- Requires: C11 or later compiler with Unicode identifier support

### Phase 2: Basic Transpiler (1-2 months)
- Hand-written lexer in C with UTF-8/NFC support
- Recursive descent parser for C-like grammar with Korean keywords
- AST -> C code generator with `#line` directives
- Support: variables, functions, control flow, basic types, structs
- Tool: Consider RE/flex + Bison, or pure hand-written

### Phase 3: Full Language (3-6 months)
- Complete C11 feature coverage
- Korean standard library wrappers
- Error messages in Korean
- VS Code extension (syntax highlighting + basic LSP)
- Documentation and tutorials in Korean

### Phase 4: Ecosystem (ongoing)
- Package manager
- Debugger integration
- Korean-language documentation generator
- Community building

### Technical Stack Recommendation
- **Transpiler language**: C (dogfooding) or Python (faster development)
- **Lexer**: Hand-written with UTF-8 support, NFC normalization
- **Parser**: Recursive descent (simpler, full control) or ANTLR4 (faster iteration)
- **Output**: Standard C11 with `#line` directives
- **Build system**: Wrapper around gcc/clang that first transpiles .한 files to .c
- **File extension**: `.한` or `.hc` (Hangul C) or `.한C`
