# 16. 자연어 문법 개선 설계서

## 1. 현황 분석

### 1.1 현재 글 언어의 자연스러움

글 언어는 이미 한국어 문법의 핵심 요소를 잘 활용하고 있다:

- **조사 기반 인자 바인딩**: `5를 3에 더하기다` -- 어순에 무관하게 조사가 역할 결정
- **어미 기반 제어흐름**: `-면`(조건), `-동안`(반복), `-다`(종결)
- **비교 동사**: `번호가 10보다 작은동안` -- 이미 자연스러운 한국어 비교문
- **멤버 접근**: `객체의 필드` -- 소유격 조사 `의` 활용
- **형변환**: `값을 정수로` -- 수단 조사 `로/으로` 활용

그러나 다음 영역에서 여전히 "프로그래밍 언어"느낌이 남아 있다:

| 현재 구문 | 자연스러운 한국어 | 문제 |
|-----------|------------------|------|
| `정수 값은 42.` | `값은 42이다.` | 타입 선언이 강제적 |
| `10을 피보나치다.` | `10의 피보나치값을 구하다.` | 함수가 동사처럼 안 읽힘 |
| `(번호 < 100)동안 { }` | `번호가 100보다 작은동안 { }` | C 스타일 괄호 비교 |
| `결과를 출력하다.` 후 `값을 계산하다.` | `결과를 출력하고 값을 계산하다.` | 문장 연결 불가 |
| `번호++다.` | `번호를 증가시키다.` | C 연산자 노출 |

### 1.2 현재 렉서/파서의 강점과 제약

**강점** (이미 구현됨):
- 형태소 분리: `결과를` → `결과` + `를`, `크거나같으면` → `크거나같은` + `면`
- 서술격 조사 `이다` 분리: `42이다` → `42` + `이` + `다`
- 비교 동사 → C 연산자 변환: `작은`, `큰`, `같은` 등 6종
- 심볼 테이블로 변수명 보호: 등록된 이름은 접사 분리 방지
- 분리 제외 사전: `길이`, `나이` 등 오분리 방지
- `보다` 조사로 비교 대상 마킹

**제약**:
- 접사 테이블이 정적 (런타임 확장 불가)
- 형태소 분리가 최대 3단계 재귀 (깊은 활용형 처리 한계)
- 타입 추론 없음 (모든 변수에 타입 명시 필수)
- 문장 간 연결 어미(`-고`, `-서`) 미활용

## 2. 개선 제안 (우선순위순)

### 등급 기준
- **영향도**: 코드의 자연스러움 향상 정도
- **난이도**: 파서/렉서 수정 범위
- **호환성**: 기존 코드 깨짐 여부

---

### 제안 1: 문장 연결 어미 `-고` (순차 실행)

**등급**: 영향도 상 / 난이도 중 / 호환성 완전

**현재**:
```글
결과를 출력하다.
번호++다.
```

**개선**:
```글
결과를 출력하고 번호++다.
```

**설명**: 한국어에서 `-고`는 두 동작을 순차적으로 연결하는 연결 어미이다. `출력하고`는 `출력하` + `고`로 분리되며, `고`는 이미 `종류_고`로 렉서에 등록되어 있다.

**파서 변경**:
- `문장파싱`에서 어미 `-고`를 만나면 현재 문장을 완료하고 다음 문장을 파싱하여 순차 블록으로 묶음
- 코드생성: `표현식1; 표현식2;` 형태로 세미콜론 분리

**구체적 파싱 흐름**:
```
"결과를 출력하고 번호++다."
 → 결과를 출력 [고] → 첫 번째 문장 완료
 → 번호++ [다] [.] → 두 번째 문장 완료
 → 블록 { 출력(결과); 번호++; }
```

**렉서 변경**: 없음 (`고`는 이미 어미로 인식)

**파서 변경 위치**: `자체호스팅/구문분석기.글`의 `문장파싱` 함수와 `표현식파싱` 후 어미 처리부

**호환성**: 현재 `-고`는 어미로 인식되지만 파서에서 무시됨. 새 기능 추가이므로 기존 코드에 영향 없음.

---

### 제안 2: 타입 추론 (`값은 42.`)

**등급**: 영향도 상 / 난이도 상 / 호환성 완전

**현재**:
```글
정수 값은 42.
실수 원주율은 3.14.
문자열 이름은 "철수".
참거짓 완료는 참.
```

**개선** (타입 명시와 공존):
```글
값은 42.               (* 정수로 추론 *)
원주율은 3.14.          (* 실수로 추론 *)
이름은 "철수".          (* 문자열로 추론 *)
완료는 참.              (* 참거짓으로 추론 *)

정수 값은 42.           (* 기존 문법도 유효 *)
```

**추론 규칙**:
| 초기값 형태 | 추론 타입 |
|------------|----------|
| 정수 리터럴 (`42`) | `정수` (int) |
| 실수 리터럴 (`3.14`) | `실수` (double) |
| 문자열 리터럴 (`"..."`) | `문자열` (char*) |
| 문자 리터럴 (`'가'`) | `문자` (char) |
| `참` / `거짓` | `참거짓` (bool) |
| `없음` | `공허 참조` (void*) |
| 함수 호출 | 함수 반환 타입 |

**파서 변경**:
- `문장파싱`에서 `식별자 + 은/는` 패턴 감지 시, 타입 없이 변수 선언으로 처리
- 현재는 `타입시작인가()` 검사를 먼저 하므로, 타입 키워드가 없으면 표현식으로 분기
- 새 분기: `식별자 + 은/는 + 값 + 다/. ` → 추론 변수 선언

**코드생성 변경**:
- 추론 변수 선언 노드에 타입 정보 없음 → 초기값 노드 종류에서 C 타입 결정
- 정수리터럴 → `int`, 실수리터럴 → `double`, 문자열리터럴 → `char*`

**제약사항**:
- 포인터 타입, 배열, 묶음(struct) 등 복합 타입은 추론 불가 — 명시 필수
- 함수 반환 타입 추론은 심볼 테이블에 함수 시그니처 저장 필요 (2단계)
- `값은 42.`에서 `값`이 기존 변수이면 대입으로 해석해야 함 → 심볼 테이블 조회 필요

**호환성**: 기존 `정수 값은 42.` 문법 그대로 유효. 새 문법은 추가 경로.

---

### 제안 3: 한글 동사형 증감/대입 (`증가시키다`, `줄이다`)

**등급**: 영향도 중 / 난이도 하 / 호환성 완전

**현재**:
```글
번호++다.
값 += 10다.
```

**개선** (추가 구문):
```글
번호를 증가시키다.          (* 번호++ *)
번호를 감소시키다.          (* 번호-- *)
값에 10을 더하다.           (* 값 += 10 *)
값에서 5를 빼다.            (* 값 -= 5 *)
```

**설명**: 이 구문은 기존 SOV 함수 호출 메커니즘을 그대로 활용한다. `증가시키다`, `감소시키다`, `더하다`, `빼다`를 코드생성기에서 내장 함수로 인식하면 된다.

**코드생성 변경** (`코드생성기.c` / `코드생성기.글`):
- `생성_호출`에서 함수명이 `증가시키`, `증가`이면 → `인자++` 생성
- `감소시키`, `감소`이면 → `인자--` 생성
- `더하`이면 → `목적지 += 값` 생성 (에 = 목적지, 을/를 = 값)
- `빼`이면 → `출처 -= 값` 생성 (에서 = 출처, 을/를 = 값)

**렉서/파서 변경**: 없음. 기존 SOV 호출 파싱이 그대로 처리.

**호환성**: 완전. 새로운 내장 함수 이름 추가일 뿐.

---

### 제안 4: `N번 반복하다` (횟수 반복)

**등급**: 영향도 중 / 난이도 중 / 호환성 완전

**현재**:
```글
반복 (정수 번호는 0: 번호 < 10: 번호++) {
    (* 본문 *)
}
```

**개선** (추가 구문):
```글
10번 반복하다 {
    (* 본문 — 암시적 카운터 없음 *)
}
```

또는 카운터 지정:
```글
번호가 10번 반복하다 {
    (* 번호: 0~9 *)
}
```

**파서 변경**:
- `문장파싱`에서 `정수리터럴 + 번 + 반복하다` 패턴 감지
- `번`은 렉서에서 식별자로 읽힘 → 파서에서 특수 처리
- `반복하다`는 이미 키워드 (`종류_반복하기`)

**코드생성**: `for (int _i = 0; _i < N; _i++) { ... }` 형태로 변환

**호환성**: 기존 `반복` / `반복하기` 문법과 공존.

---

### 제안 5: 서술형 대입 `이다` 강화

**등급**: 영향도 중 / 난이도 하 / 호환성 완전

**현재** (이미 작동하는 부분):
```글
정수 값은 42이다.     (* 작동 — 이+다 분리 *)
정수 번호는 0이다.    (* 작동 *)
```

**문제점**: 재대입에서 `이다`가 자연스럽지 않음:
```글
값 = 100다.           (* 현재: C 스타일 대입 *)
```

**개선**:
```글
값은 100이다.          (* 재대입도 서술형으로 *)
```

**파서 변경**:
- `문장파싱`에서 `식별자 + 은/는 + 표현식 + 이다/.` 패턴이 나올 때:
  - 해당 식별자가 심볼 테이블에 등록된 변수이면 → 대입문으로 처리
  - 미등록이면 → 추론 변수 선언 (제안 2)

**현재 동작과의 차이**: 현재 `값은 100이다.`는 `값`이 타입이 아니므로 표현식으로 파싱됨. SOV 호출로 해석을 시도하다 실패하거나, `값`을 함수로 찾게 됨. 이를 대입문으로 인식하는 별도 경로가 필요.

**코드생성**: `값 = 100;`

**호환성**: 기존 `값 = 100다.` 문법도 유지.

---

### 제안 6: 묶음 정의에 `이다` 마침

**등급**: 영향도 하 / 난이도 하 / 호환성 완전

**현재**:
```글
묶음 학생은
    문자열 성명,
    정수 연령,
    정수 점수.
```

**개선** (선택적):
```글
묶음 학생은
    문자열 성명,
    정수 연령,
    정수 점수이다.
```

**설명**: 구조체 정의의 마지막 필드 뒤에 `이다`를 허용하면 "학생은 ... 이다"라는 완전한 문장이 된다. 마침표만으로도 작동하되, `이다.`도 허용.

**파서 변경**: `묶음정의_파싱`에서 마지막 필드 뒤의 `이` + `다` + `.` 패턴을 마침 표현으로 인식.

---

### 제안 7: 함수를 동사로 더 자연스럽게

**등급**: 영향도 중 / 난이도 상 / 호환성 완전

**현재**:
```글
[정수 수를 피보나치]는 → 정수 { ... }
10을 피보나치다.
```

**문제**: `피보나치`가 명사인지 동사인지 모호. 한국어에서 함수는 동사여야 자연스럽다.

**개선 방향 A — 동사형 함수 이름**:
```글
[정수 수를 피보나치_구하기]는 → 정수 { ... }
10의 피보나치를 구하다.
```

**개선 방향 B — `하다` 접미 자동 인식**:
```글
[정수 수를 계산하기]는 → 정수 { ... }
10을 계산하다.     (* "계산하" → "계산" + 동사화 "하" 제거 → 함수 "계산" 호출 *)
```

**현재 구현 상태**: 렉서의 부트스트랩 C 코드에 이미 `동사형` 처리가 있다 (`형태소분석기.c` 547-562행). `~하다`/`~해`로 끝나는 단어에서 `하`/`해`를 분리하고 어근만 추출하는 로직이 구현되어 있다. 그러나 자체호스팅 렉서(`형태소분석기.글`)에는 이 로직이 빠져 있다.

**필요한 작업**: 자체호스팅 렉서에 동사형 `하`/`해` 분리 로직 이식.

**호환성**: `하다`가 붙지 않는 기존 호출 (`피보나치다`)도 그대로 작동.

---

### 제안 8: `보다` + 비교동사의 관형사 활용 확장

**등급**: 영향도 하 / 난이도 하 / 호환성 완전

**현재 (이미 작동)**:
```글
번호가 10보다 작은동안 { ... }    (* while *)
점수가 90보다 크면 { ... }        (* if *)
```

**추가 가능한 관형사형**:
```글
값이 10보다 큰 경우 { ... }       (* switch-like *)
값이 10보다 작지않으면 { ... }    (* !(값 < 10) → 값 >= 10 *)
```

**부정형 처리**: `작지않으면` → `작지않` + `으면` → `!(작)` + 조건
- 렉서에 `지않` 접사 추가 (부정 관형사형)
- 코드생성에서 비교동사 앞에 `!` 추가

**이 제안은 낮은 우선순위** — 현재 비교 동사 체계가 이미 잘 작동하며, 부정형은 `값 >= 10이면`으로 충분히 표현 가능.

## 4. 개선 전후 코드 비교

### 4.1 피보나치 함수 (현재)

```글
[정수 번을 피보나치]는 → 정수 {
    번이 1보다 작거나같으면
        반환 번다.
    정수 앞은 번 - 1을 피보나치다.
    정수 뒤는 번 - 2를 피보나치다.
    반환 앞 + 뒤다.
}

[시작하기]는 {
    정수 번호는 0이다.
    번호가 10보다 작은동안 {
        (번호를 피보나치)를 출력하다.
        번호++다.
    }
}
```

### 4.2 피보나치 함수 (제안 적용 후)

```글
[정수 번을 피보나치]는 → 정수 {
    번이 1보다 작거나같으면
        반환 번다.
    앞은 번 - 1을 피보나치다.          (* 타입 추론 *)
    뒤는 번 - 2를 피보나치다.          (* 타입 추론 *)
    반환 앞 + 뒤다.
}

[시작하기]는 {
    번호는 0이다.                       (* 타입 추론 *)
    번호가 10보다 작은동안 {
        (번호를 피보나치)를 출력하고    (* -고 연결 *)
        번호를 증가시키다.              (* 동사형 증감 *)
    }
}
```

### 4.3 동적 배열 (현재)

```글
[동적배열_테스트]는 {
    정수 용량 = 4.
    정수 개수 = 0.
    정수 참조 배열 = (크기(정수) * 용량)을 할당다.

    정수 번호 = 0.
    번호가 8보다 작은동안 {
        개수가 용량보다 크거나같으면 {
            용량 = 용량 * 2.
            배열 = 배열을 (크기(정수) * 용량)으로 재할당다.
        }
        배열[개수] = (번호 + 1) * 11.
        개수++다.
        번호++다.
    }
}
```

### 4.4 동적 배열 (제안 적용 후)

```글
[동적배열_테스트]는 {
    용량은 4이다.                                    (* 타입 추론 *)
    개수는 0이다.                                    (* 타입 추론 *)
    정수 참조 배열 = (크기(정수) * 용량)을 할당다.   (* 포인터는 명시 *)

    번호는 0이다.                                    (* 타입 추론 *)
    번호가 8보다 작은동안 {
        개수가 용량보다 크거나같으면 {
            용량에 2를 곱하다.                        (* 동사형 대입 — 미래 *)
            배열 = 배열을 (크기(정수) * 용량)으로 재할당다.
        }
        배열[개수] = (번호 + 1) * 11.
        개수를 증가시키고 번호를 증가시키다.          (* -고 연결 + 동사형 *)
    }
}
```

## 5. 기술 상세: 주요 변경 지점

### 5.1 `-고` 연결 어미 구현

**파일**: `자체호스팅/구문분석기.글`

현재 `어미소비` 함수에서 `-고`를 인식하지만, `문장파싱`에서는 `-고` 이후 처리가 없다.

변경 방향:
```
문장파싱() {
    ...
    표현식 = 표현식파싱()
    어미 = 어미소비()

    어미 == 어미_고 이면 {
        (* 현재 문장 완료, 다음 문장 파싱 *)
        첫문장 = 현재 표현식을 문장으로 포장
        둘째문장 = 문장파싱()   (* 재귀 *)
        반환 순차블록(첫문장, 둘째문장)
    }
    어미 == 어미_다 이면 {
        기대(마침표)
        반환 표현식을 문장으로
    }
}
```

**C 코드생성**: 각 문장을 세미콜론으로 분리하여 출력. 블록으로 감쌀 필요 없음.

### 5.2 동사형 내장 함수

**파일**: `자체호스팅/코드생성기.글` (또는 `부트스트랩/src/코드생성기.c`)

`생성_호출` 함수에서 함수 이름을 확인하는 부분에 추가:

```
함수이름 == "증가시키" 또는 "증가" 이면:
    인자(을/를 역할) → "인자++"

함수이름 == "감소시키" 또는 "감소" 이면:
    인자(을/를 역할) → "인자--"

함수이름 == "더하" 이면:
    목적지(에 역할) += 값(을/를 역할)

함수이름 == "빼" 이면:
    출처(에서 역할) -= 값(을/를 역할)
```

이는 기존 `비교동사_연산자` 패턴과 동일한 접근법이다.

### 5.3 타입 추론 변수 선언

**파일**: `자체호스팅/구문분석기.글`의 `문장파싱`

현재 흐름:
```
문장파싱:
  타입시작인가? → 변수선언파싱
  아니면       → 표현식파싱 → 어미/대입 처리
```

변경 흐름:
```
문장파싱:
  타입시작인가? → 변수선언파싱
  식별자 + (은/는)? → 심볼테이블 확인
    등록된 변수 → 서술형 재대입 (값은 100이다.)
    미등록      → 추론 변수 선언 (값은 42.)
  아니면       → 표현식파싱 → 어미/대입 처리
```

**AST 노드 변경**: 변수 선언 노드에 `타입 = -1` (추론 플래그). 코드생성 시 초기값 종류에서 타입 결정.

## 6. 호환성 보장 원칙

1. **기존 문법 100% 유지**: 모든 개선은 새로운 대안 경로 추가. 기존 `.글` 파일은 수정 없이 컴파일됨.
2. **자체호스팅 무결성**: 컴파일러 자체 소스(`자체호스팅/*.글`)는 기존 문법 사용. 새 문법은 사용자 코드에서 선택적 사용.
3. **단계적 적용**: 각 제안은 독립적으로 구현 가능. 하나의 제안이 다른 제안에 의존하지 않음.
4. **SHA256 고정점**: 새 기능 추가 후에도 기존 컴파일러 소스의 자체호스팅 결과는 동일해야 함 (새 기능을 컴파일러 소스에서 사용하지 않는 한).

## 7. 불채택 아이디어와 근거

### 7.1 완전한 자연어 파싱

```
(* 이상적이지만 비현실적 *)
만약 사용자의 나이가 18세 이상이면 성인으로 분류하다.
```

**불채택 근거**: 한국어의 모호성 해소가 매우 어려움. "이상"이 `이상`(abnormal)인지 `이상`(more than)인지, 문맥에 따라 다름. 프로그래밍 언어는 결정적 파싱이 필수.

### 7.2 암시적 주어 생략

```
(* 한국어에서 자연스럽지만 위험 *)
10을 계산하다.          (* 누가 계산? 무엇의? *)
출력하다.               (* 무엇을? *)
```

**불채택 근거**: 프로그래밍에서 암시적 인자는 버그의 원인. 글 언어의 조사 시스템이 인자를 명시하는 것이 오히려 강점.

### 7.3 `의` 조사를 함수 호출에 활용

```
(* 제안됨 *)
10의 피보나치값.
배열의 길이.
```

**불채택 근거**: `의`는 이미 멤버 접근(`객체의 필드`)으로 사용 중. 함수 호출에도 사용하면 `값의 계산`이 멤버 접근인지 함수 호출인지 모호해짐.

### 7.4 과거 시제 어미

```
(* 한국어 과거형 *)
값을 계산했다.          (* 이미 완료됨을 강조 *)
파일을 열었다.
```

**불채택 근거**: 프로그래밍 언어에서 시제 구분은 의미 없음. 모든 코드는 "실행 시점"에서 현재형. 복잡성만 증가.

## 8. 구현 완료 (2026-03-21)

### 구현된 기능

1. **`결과를 반환하다.`** — SOV 반환문 (제안 7 변형)
   - 파서의 `SOV_호출파싱`에서 동사 위치에 `반환` 키워드/식별자 감지 시 반환문 생성
   - 기존 `반환 결과.` 문법과 공존
   - 부트스트랩 C 파서 + 자체호스팅 글 파서 양쪽 모두 구현

2. **`번호를 증가하다.`** / **`번호를 감소하다.`** — 동사형 증감 (제안 3)
   - 파서의 `SOV_호출파싱`에서 동사 위치에 `증가`/`감소` 식별자 감지 시 증감문 생성
   - 기존 `번호++다.` 문법과 공존

3. **`-고` 연결 어미** — 이전에 구현 완료 (제안 1)
   - `출력하고 증가하다.` → 두 문장 순차 실행

4. **`하다` 동사형 호출** — 이전에 구현 완료 (제안 7)
   - 렉서의 동사형 분리: `계산하다` → `계산` + `다`

### 기술 노트

- 동사형 분리("반환하다" → "반환"+"다")에서 어근이 항상 `종류_식별자`로 나옴
- 따라서 SOV 동사 위치에서 `종류_반환` 뿐 아니라 텍스트가 "반환"인 `종류_식별자`도 확인 필요
- 문장 파싱에서 `표현식파싱` 반환값이 `노드_반환문`이면 어미/마침표 소비 후 직접 반환

### 미구현 기능

- 제안 2: 타입 추론 — 심볼 테이블 확장 필요
- 제안 4: `N번 반복하다` — 횟수 반복 구문
- 제안 5: 서술형 재대입 `값은 100이다.`

## 9. 결론

글 언어는 이미 한국어 문법을 프로그래밍에 성공적으로 적용한 사례이다. 현재의 비교 동사 시스템(`보다 + 작은/큰/같은 + 동안/면`)은 자연스러운 한국어 문장 구조를 잘 구현하고 있다.

구현 완료된 개선:
1. **동사형 증감** (`번호를 증가하다.`) — 파서 SOV 동사 인식
2. **SOV 반환** (`결과를 반환하다.`) — 파서 SOV 동사 인식
3. **`-고` 연결 어미** (`출력하고 증가하다.`) — 파서 어미 분기
4. **`하다` 동사형 호출** (`계산하다.`) — 렉서 동사어근 분리

향후 가장 효과적인 추가 개선:
1. **타입 추론** (가장 큰 변화이지만 코드 간결성 대폭 향상)
2. **`N번 반복하다`** (자연스러운 반복 구문)
