# 한글 프로그래밍 언어 - 아키텍처 설계서

## 언어명: 글 (Geul)
> "글"은 한국어로 "writing/text"를 의미하며, 코드도 결국 글이라는 철학을 담음
> 파일 확장자: `.글`

## 2. 각 단계 상세 설계

### 2.1 전처리기 (전처리)

**입력**: .글 소스파일 (UTF-8)
**출력**: 전처리 완료된 소스 텍스트

**기능**:
1. **유니코드 정규화**: 모든 입력을 NFC로 정규화 (macOS NFD 문제 해결)
2. **파일 포함**: `포함 "파일경로"` → 해당 파일 내용 삽입
3. **상수 정의**: `정의 이름 값` → 텍스트 치환
4. **조건부 컴파일**: `#조건 ... #아니면 ... #끝` (플랫폼별 분기)

```
포함 "표준/입출력"
포함 "표준/수학"

정의 최대크기 1024
정의 버전 "1.0.0"
```

### 2.2 형태소 분석기 (어휘 분석 / Lexer)

**입력**: 전처리된 UTF-8 텍스트
**출력**: 토큰 스트림

이것이 가장 핵심적인 단계. 일반 프로그래밍 언어의 렉서와 근본적으로 다름.

#### 2.2.1 토큰 종류

```
토큰_종류 {
    // 리터럴
    정수_리터럴,        // 42, 0xFF, 0b1010
    실수_리터럴,        // 3.14, 1.0e-5
    문자열_리터럴,      // "안녕하세요"
    문자_리터럴,        // '가'

    // 식별자 (사용자 정의 이름)
    식별자,             // 사과가격, 합계, 카운터

    // 타입 키워드
    정수, 실수, 문자, 문자열, 참거짓, 공허,
    긴정수, 짧은정수, 부호없는,

    // 조사 (역할 표지)
    조사_을, 조사_를,   // 목적어
    조사_이, 조사_가,   // 주어
    조사_은, 조사_는,   // 주제
    조사_에,            // 목적지/위치
    조사_로, 조사_으로, // 수단/방법
    조사_에서,          // 출처
    조사_부터,          // 시작
    조사_까지,          // 종점
    조사_와, 조사_과,   // 공동
    조사_의,            // 소유/멤버접근
    조사_보다,          // 비교

    // 어미 (제어흐름)
    어미_다,            // 서술/종결  → 문장 끝
    어미_면,            // 조건      → if
    어미_고,            // 접속      → 순차(;)
    어미_서,            // 인과/순차  → 파이프라인
    어미_거나,          // 선택      → or
    어미_지만,          // 양보      → fallback
    어미_도록,          // 목적      → assert/ensure
    어미_때,            // 시간      → when/event
    어미_동안,          // 지속      → while
    어미_라,            // 명령      → execute
    어미_자,            // 제안      → spawn
    어미_까,            // 의문      → test/query
    어미_며,            // 동시      → parallel

    // 구조 키워드
    묶음,              // struct
    나열,              // enum
    함수,              // function definition marker
    반환,              // return
    탈출,              // break
    계속,              // continue
    크기,              // sizeof

    // 구분자
    여는괄호, 닫는괄호,     // ( )
    여는중괄호, 닫는중괄호, // { }
    여는대괄호, 닫는대괄호, // [ ]
    쉼표,                   // ,
    마침표,                 // .  (문장 종결자)

    // 연산자
    더하기, 빼기, 곱하기, 나누기, 나머지,
    같다, 다르다, 크다, 작다, 크거나같다, 작거나같다,
    그리고, 또는, 아닌,
    비트와, 비트또는, 비트배타, 비트아닌,
    왼쪽이동, 오른쪽이동,
    화살표,            // →  (함수 반환타입, 포인터 역참조)
    참조,              // &  (주소 참조)

    // 특수
    줄끝,
    파일끝,
}
```

#### 2.2.2 형태소 분리 로직

핵심 과제: `사과가격을` 에서 `사과가격` + `을`(조사)을 분리해야 함.

```
형태소분리 알고리즘:

1. 한글 토큰을 읽음 (연속된 한글 음절)
2. 뒤에서부터 알려진 조사/어미 패턴 매칭 시도
   - 2음절 조사 먼저: "에서", "부터", "까지", "보다", "으로", "에게"
   - 1음절 조사: "을", "를", "이", "가", "은", "는", "에", "로", "와", "과", "의"
   - 어미: "면", "고", "서", "다", "라", "까", "며"
   - 복합어미: "거나", "지만", "도록", "동안"
3. 분리된 어근이 알려진 키워드/식별자인지 확인
4. 모호성 해결:
   - "가"가 식별자의 일부인지 조사인지 → 심볼 테이블 참조
   - 미등록 식별자면 가장 긴 매칭 우선
   - 등록된 식별자면 정확한 매칭 우선
```

**모호성 해결 전략 (핵심)**:

한글의 다양한 표현이 문제가 되지 않도록 **언어가 허용하는 조사/어미를 명확히 제한**:

```
규칙 1: 식별자는 반드시 조사와 분리하여 작성
        → "사과가격 을" (O)   "사과가격을" (O, 자동 분리)
        방법: 공백 있으면 분리 확정, 없으면 뒤에서 조사 패턴 매칭

규칙 2: 조사는 정해진 목록만 인식
        → 을/를, 이/가, 은/는, 에, 로/으로, 에서, 부터, 까지, 와/과, 의, 보다

규칙 3: 어미는 동사/함수 뒤에만 인식
        → 출력하다 (O), 출력하면 (O), 출력하고 (O)
        → 동사형: 어근 + "하" + 어미

규칙 4: 모호한 경우 공백으로 명시적 분리
        → 변수명이 "나"이고 조사가 "를"이면: "나 를" 또는 "나를" 둘 다 허용
        → 변수명이 "나를"이면: 반드시 "나를 을" 처럼 공백+조사
```

### 2.3 구문 분석기 (Parser)

**입력**: 토큰 스트림
**출력**: 추상 구문 트리 (AST)

#### 2.3.1 문법 (EBNF 형식)

```ebnf
프로그램      = { 최상위선언 }

최상위선언    = 함수정의 | 묶음정의 | 나열정의 | 전역변수선언

(* === 함수 정의 === *)
함수정의      = '[' 매개변수목록 동사원형 ']' '는' 반환타입? 블록
매개변수목록  = { 타입 이름 조사 }
동사원형      = 식별자                    (* 예: "더하기", "정렬하기" *)
반환타입      = '→' 타입
블록          = 들여쓰기된_문장들 | '{' 문장들 '}'

(* === 문장 === *)
문장          = 선언문 | 대입문 | 조건문 | 반복문 | 호출문 | 반환문 | 파이프라인

(* === 선언 === *)
선언문        = 타입 이름 '는' 표현식 '다.'
             | 타입 이름 '는' 표현식 '이다.'

(* === 대입 === *)
대입문        = 표현식 '을/를' 이름 '에' 동사_대입
동사_대입     = '넣다.' | '넣고' | '넣어서' | ...

(* === 조건문 === *)
조건문        = 표현식_조건 블록 아니면절?
표현식_조건   = 표현식 '면'                (* -면 어미가 if 역할 *)
아니면절      = '아니면' 블록
             | '아니면' 조건문             (* else if 체이닝 *)

(* === 반복문 === *)
반복문        = 표현식 '동안' 블록         (* while *)
             | 이름 '이/가' 범위 '동안' 블록  (* for range *)
범위          = 표현식 '부터' 표현식 '까지'

(* === 호출문 (SOV 구조) === *)
호출문        = { 인자 조사 } 동사 어미
인자          = 표현식
동사          = 식별자 '하'               (* 동사 어근 *)
어미          = '다' | '고' | '서' | '면' | '라' | '거나' | ...

(* === 파이프라인 (-어서/-아서 체이닝) === *)
파이프라인    = 표현식 { '어서'|'아서' 동사 } 동사 어미_종결

(* === 표현식 === *)
표현식        = 비교식
비교식        = 이름 '이/가' 표현식 '보다' 비교동사
             | 산술식 비교연산자 산술식
산술식        = 항 { ('더하기'|'빼기') 항 }
항            = 인수 { ('곱하기'|'나누기') 인수 }
인수          = 정수_리터럴 | 실수_리터럴 | 문자열_리터럴
             | 이름 | 이름 '의' 이름       (* 멤버 접근 *)
             | '(' 표현식 ')'
             | 이름 '[' 표현식 ']'         (* 배열 접근 *)

(* === 타입 === *)
타입          = '정수' | '실수' | '문자' | '문자열' | '참거짓' | '공허'
             | '긴' 타입 | '짧은' 타입 | '부호없는' 타입
             | 타입 '참조'                 (* 포인터 *)
             | 타입 '[' 정수 ']'           (* 배열 *)
             | 묶음이름                    (* 구조체 *)
```

#### 2.3.2 AST 노드 구조

```
AST노드 종류:
├── 프로그램노드
│   └── 최상위선언[]
├── 함수정의노드
│   ├── 이름: 문자열
│   ├── 매개변수: [(타입, 이름, 역할조사)]
│   ├── 반환타입: 타입노드
│   ├── 본문: 블록노드
│   └── 위치: (파일, 줄, 열)
├── 변수선언노드
│   ├── 타입: 타입노드
│   ├── 이름: 문자열
│   ├── 초기값: 표현식노드?
│   └── 위치
├── 조건노드
│   ├── 조건: 표현식노드
│   ├── 참블록: 블록노드
│   ├── 거짓블록: 블록노드?
│   └── 위치
├── 반복노드
│   ├── 종류: 동안반복 | 범위반복
│   ├── 조건/범위: 표현식노드
│   ├── 본문: 블록노드
│   └── 위치
├── 호출노드 (SOV 구조)
│   ├── 동사: 문자열
│   ├── 인자: [(표현식노드, 역할조사)]  ← 조사가 역할 결정
│   ├── 연결어미: 어미종류           ← 다음 문장과의 관계
│   └── 위치
├── 파이프라인노드
│   ├── 시작값: 표현식노드
│   ├── 단계: [동사명]
│   ├── 종결어미: 어미종류
│   └── 위치
├── 표현식노드
│   ├── 이항연산노드 { 왼쪽, 연산자, 오른쪽 }
│   ├── 리터럴노드 { 종류, 값 }
│   ├── 식별자노드 { 이름 }
│   ├── 멤버접근노드 { 대상, 멤버 }  ← "의" 조사
│   ├── 배열접근노드 { 배열, 인덱스 }
│   └── 위치
└── 타입노드
    ├── 기본타입 { 종류 }
    ├── 참조타입 { 대상타입 }
    ├── 배열타입 { 원소타입, 크기 }
    └── 묶음타입 { 이름 }
```

### 2.4 의미 분석기

**입력**: AST
**출력**: 타입이 확정된 AST + 심볼 테이블

**수행 작업**:

```
1. 심볼 테이블 구축
   - 스코프 체인 관리 (주제-평언 구조의 '은/는' 스코프 포함)
   - 이름 해석: 식별자 → 선언 위치 연결
   - 중복 선언 검사

2. 타입 검사
   - 대입 호환성 검사 (정수에 실수 대입 등)
   - 함수 호출 시 조사-매개변수 역할 매칭
     예: [정수 가를 정수 나에 더하기] 정의에서
         "5를 3에 더하라" → 가=5(를=목적어), 나=3(에=목적지)
   - 연산자 타입 호환성
   - 암시적 형변환 규칙 정의

3. 조사 역할 해석
   - 함수 매개변수의 조사와 호출부의 조사 매칭
   - 같은 조사의 인자가 여러 개면 오류
   - 조사가 생략된 경우 위치 기반 폴백

4. 어미 연결 분석
   - -고 (순차): 다음 문장이 이어짐 확인
   - -면 (조건): 블록이 따라오는지 확인
   - -어서 (파이프라인): 반환값이 다음 동사의 입력과 타입 호환 확인
   - -다 (종결): 문장 끝 확인

5. 암시적 컨텍스트 해석 (주어 생략)
   - '은/는'으로 설정된 주제 추적
   - 주어/목적어 생략 시 현재 주제로 대체
   - 주제 스코프 경계 확인
```

### 2.5 중간표현 생성기 (글IR)

AST와 LLVM IR 사이의 자체 중간표현. 한글 문법의 고수준 의미를 보존하면서
LLVM으로 변환하기 쉬운 형태.

**글IR이 필요한 이유**:
- 조사 기반 인자 바인딩을 위치 기반으로 정규화
- 파이프라인(-어서 체인)을 순차 호출로 변환
- 암시적 컨텍스트(주어 생략)를 명시적 인자로 변환
- 한글 고유 최적화 수행 (중복 조사 제거 등)

```
글IR 명령어:
├── 함수정의(이름, 매개변수[], 반환타입, 본문블록)
├── 블록(라벨, 명령어[])
├── 변수할당(이름, 타입, 값)
├── 적재(대상, 타입)
├── 저장(값, 대상)
├── 호출(함수, 인자[])
├── 조건분기(조건, 참라벨, 거짓라벨)
├── 무조건분기(라벨)
├── 반환(값?)
├── 이항연산(연산, 왼쪽, 오른쪽)
├── 형변환(값, 원본타입, 대상타입)
├── 멤버접근(묶음값, 멤버인덱스)
├── 배열접근(배열, 인덱스)
└── 파이(값들[])        // SSA PHI 노드
```

### 2.6 LLVM IR 생성기

**입력**: 글IR
**출력**: LLVM IR (Module)

```
변환 규칙:

글IR                    →  LLVM IR
──────────────────────────────────────────
정수                    →  i32 (또는 i64)
실수                    →  double
문자                    →  i8
참거짓                  →  i1
참조(T)                 →  ptr (opaque pointer)
배열(T, N)              →  [N x T]
묶음 { 필드들 }        →  %struct.이름 = type { 필드타입들 }

변수할당                →  %name = alloca T
적재                    →  %tmp = load T, ptr %name
저장                    →  store T %val, ptr %name
호출                    →  %ret = call T @func(args...)
조건분기                →  br i1 %cond, label %true, label %false
이항연산(+)             →  %tmp = add i32 %l, %r
이항연산(==)            →  %tmp = icmp eq i32 %l, %r
반환                    →  ret T %val

디버그 정보:
- DIBuilder로 모든 함수/변수/타입에 DWARF 메타데이터 부착
- DILocation으로 모든 명령어에 소스 위치(글 파일의 줄/열) 매핑
- DW_LANG_C 사용 (디버거 호환성)
```

### 2.7 LLVM 최적화

```
최적화 파이프라인 (레벨별):

-ㅇ0 (없음): 최적화 없음, 디버깅용
-ㅇ1 (기본): mem2reg, 상수 접기, 죽은 코드 제거
-ㅇ2 (표준): + 인라이닝, 루프 최적화, GVN, LICM
-ㅇ3 (최대): + 벡터화, 루프 풀기, 공격적 인라이닝
-ㅇ크 (크기): 코드 크기 최소화
```

### 2.8 코드 생성 → 오브젝트 → 실행파일

```
LLVM 코드 생성:
1. LLVM TargetMachine 생성 (타겟 트리플 기반)
2. PassManager에 코드생성 패스 추가
3. MC 레이어로 오브젝트 파일 생성

타겟 트리플:
- x86_64-unknown-linux-gnu    (리눅스)
- x86_64-pc-windows-msvc      (윈도우)
- x86_64-apple-darwin          (맥)
- aarch64-unknown-linux-gnu    (ARM64 리눅스)

링킹:
LLD로 오브젝트 파일 + 글 런타임 라이브러리 링크
→ 네이티브 실행파일 출력
```

## 3. 런타임 라이브러리 (글런타임)

### 3.1 시작점 (_시작)

C의 crt0/crt1에 해당. 플랫폼별 어셈블리로 작성:

```
_시작 (리눅스 x86-64):
1. 스택에서 argc, argv, envp 추출
2. 글 런타임 초기화
   - 힙 초기화 (mmap으로 초기 힙 영역 확보)
   - 표준 입출력 초기화
   - 전역 변수 초기화 (.init_array 실행)
3. 사용자의 시작함수 호출
4. 종료 코드로 exit 시스콜 호출

_시작 (윈도우 x86-64):
1. kernel32.dll에서 GetCommandLineW, ExitProcess 등 함수 포인터 확보
2. 명령줄 파싱하여 argc, argv 구성
3. 글 런타임 초기화
4. 사용자의 시작함수 호출
5. ExitProcess 호출
```

### 3.2 메모리 관리

```
리눅스: mmap 시스콜 직접 사용
  - 할당(크기) → mmap(NULL, 크기, PROT_READ|PROT_WRITE,
                       MAP_PRIVATE|MAP_ANONYMOUS, -1, 0)
  - 해제(주소, 크기) → munmap(주소, 크기)
  - 내부: 프리리스트 기반 할당자 구현
    - 작은 블록 (< 4KB): 프리리스트에서 할당
    - 큰 블록 (>= 4KB): 직접 mmap

윈도우: VirtualAlloc/VirtualFree 사용
  - kernel32.dll의 VirtualAlloc 호출
  - 동일한 프리리스트 할당자 로직
```

### 3.3 입출력

```
리눅스:
  출력(문자열) → write(1, 문자열, 길이)  [시스콜 1번]
  입력(버퍼, 크기) → read(0, 버퍼, 크기) [시스콜 0번]
  파일열기(경로) → open(경로, 플래그)     [시스콜 2번]

윈도우:
  출력(문자열) → WriteFile(GetStdHandle(-11), ...)
  입력(버퍼, 크기) → ReadFile(GetStdHandle(-10), ...)
  파일열기(경로) → CreateFileW(경로, ...)
```

### 3.4 표준 라이브러리 구조

```
표준/
├── 입출력.글        # 출력, 입력, 파일열기, 파일읽기, 파일쓰기, 파일닫기
├── 기억.글          # 할당, 해제, 재할당, 복사, 채우기
├── 문자열.글        # 길이, 합치기, 비교, 찾기, 자르기, 바꾸기
├── 수학.글          # 절대값, 제곱근, 거듭제곱, 삼각함수
├── 체계.글          # 종료, 인자, 환경변수, 시간
└── 모음.글          # 배열 유틸리티, 정렬, 탐색
```

```
단계 0 (부트스트랩 컴파일러):
  - C로 작성된 "글" 컴파일러
  - LLVM C API 사용
  - 언어 기능의 핵심 부분집합만 지원
  - 이것으로 단계 1을 컴파일

  - "글"로 작성된 "글" 컴파일러
  - 단계 0 컴파일러로 컴파일
  - 언어의 전체 기능 지원

  - 단계 1 바이너리로 자기 자신을 컴파일
  - 단계 1과 동일한 바이너리 생성 확인
  - 이제 C에 의존하지 않는 독립 언어

단계 3 (검증):
  - 단계 2 바이너리로 다시 컴파일
  - 단계 2와 비트 동일 확인 → 자기 호스팅 완료
```
