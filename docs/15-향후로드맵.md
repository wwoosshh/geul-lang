# 글 프로그래밍 언어 — 향후 로드맵

## 현재 상태 (2026-03-20 기준)

### 달성한 것
- 자체호스팅 컴파일러 (SHA256 고정점)
- C 트랜스파일러 (.글 → C → 네이티브)
- 네이티브 백엔드 (.글 → x86-64 기계어 → PE 직접 생성, 외부 도구 0개)
- VSCode 확장 (구문 강조, F5 실행, 에러 진단)
- 표준 라이브러리 8개 모듈, 62개 함수
- 종합 벤치마크 시스템 (6개 언어 비교)
- 최적화: 피프홀 + 레지스터 핀닝 (소수판별에서 C(/O2)보다 40% 빠름)

### 핵심 수치
- 컴파일러 전체 ~9,150줄 (C 부트스트랩 + 글 자체호스팅 + 네이티브 백엔드)
- 네이티브 성능: 재귀 fib(40) C 대비 1.7x, 소수 체 C 대비 0.6x
- 지원 기능: 정수/실수 산술, 배열, 재귀, 조건문, 반복문, printf, malloc/free, QPC 타이밍
- 임포트: 31개 msvcrt + 5개 kernel32 함수

## 로드맵 단계

### P1: 언어 기능 보강 (단기)

#### P1-1: 구조체 실전 검증 및 수정
- **현황**: 기계어 인코더(필드적재/필드저장) 구현됨, IR 변환기에 필드저장 추가됨. 실제 프로그램으로 미검증.
- **작업**: 구조체 정의 → 인스턴스 생성 → 필드 읽기/쓰기 → 포인터 멤버 접근(→) 테스트
- **검증 프로그램**:
```글
묶음 점은 정수 좌표_엑스, 정수 좌표_와이.

[시작하기]는 -> 정수 {
    점 위치.
    위치.좌표_엑스 = 10.
    위치.좌표_와이 = 20.
    (* 필드 읽기 + 출력 *)
}
```
- **예상 이슈**: 필드 오프셋 계산, 구조체 크기 산정, 포인터 멤버 접근 기계어
- **파일**: `자체호스팅/PE생성기.글` (명령_필드적재/저장), `자체호스팅/IR변환기.글`

#### P1-2: qword 배열 (정수/포인터 배열)
- **현황**: 바이트 배열(char[]) 작동 확인. 정수 배열(int[], long long[])은 원소 크기 정보가 IR에 전달되지 않아 바이트로 접근됨.
- **작업**: IR 변환기에서 배열 원소 크기를 `명령정수값풀`에 기록. 타입 노드에서 원소 크기 추출.
- **파일**: `자체호스팅/IR변환기.글` (노드_배열접근 처리), `자체호스팅/PE생성기.글` (이미 크기별 분기 있음)

#### P1-3: for 루프 네이티브 지원
- **현황**: `반복 (초기: 조건: 증감) { }` 구문이 C 트랜스파일에서는 작동하지만 네이티브 백엔드의 IR 변환에서 증감 부분이 미완.
- **작업**: IR변환기에서 for 루프의 증감 표현식이 루프 본문 끝에 배치되도록 수정.
- **파일**: `자체호스팅/IR변환기.글`

#### P1-4: 에러 메시지 개선
- **현황**: `줄:열: 오류: 메시지` 형식. 기대 토큰/받은 토큰 정보 부족.
- **작업**: 파서 에러 시 `예상: '}'이지만 '식별자'가 있습니다` 같은 상세 메시지 추가.
- **파일**: `자체호스팅/구문분석기.글`

---

### P2: 개발 생태계 (중기)

#### P2-1: 다중 파일 컴파일
- **현황**: 단일 파일 + `포함`(#include) 방식. 대규모 프로젝트에서 전체 재컴파일 필요.
- **목표**: `.글` 파일별 독립 컴파일 → `.obj` → 링크
- **설계 방향**:
  1. 글 헤더 파일 (`.글ㅎ` 또는 `.글머리`) — 함수 시그니처, 타입 선언만 포함
  2. 각 `.글` 파일을 독립 컴파일하여 `.obj` (COFF) 생성
  3. 내장 링커로 여러 `.obj` + 임포트 → PE
- **대안**: 현재 `포함` 방식 유지하되, 변경된 파일만 재처리하는 증분 빌드
- **예상 작업량**: 상 (COFF 포맷 생성 또는 증분 빌드 캐시)

#### P2-2: 네이티브 표준 라이브러리
- **현황**: 표준 라이브러리가 C 함수 래핑 (`외부` 선언으로 msvcrt.dll 호출)
- **목표**: 글 표준 라이브러리를 네이티브 컴파일러에서 직접 사용 가능하게
- **설계 방향**:
  1. `표준/기본.글`을 네이티브 컴파일러에서 `포함` 지원
  2. 표준 라이브러리의 `외부` 선언이 PE 임포트 테이블에 자동 추가되도록
  3. 사용자가 `외부` 선언 없이 `"안녕\n"을 쓰기다.`처럼 호출 가능
- **전제조건**: P2-1 (다중 파일) 또는 포함 전처리기 확장

#### P2-3: LSP 서버 (Language Server Protocol)
- **현황**: VSCode 확장에 구문 강조와 에러 진단만 있음. 자동완성, 정의 이동 없음.
- **목표**: 글 LSP 서버를 글 언어로 작성
- **기능 우선순위**:
  1. 실시간 에러 진단 (파싱 에러)
  2. 자동완성 (키워드 + 선언된 변수/함수)
  3. 정의로 이동
  4. 호버 정보 (타입, 함수 시그니처)
- **구현**: 글로 작성된 LSP 서버 → 네이티브 컴파일 → .exe → VSCode가 프로세스로 실행
- **예상 작업량**: 상

#### P2-4: 패키지 관리자
- **현황**: 없음. 모든 코드가 단일 프로젝트 내.
- **목표**: `글꾸러미` (패키지) 시스템
- **설계 방향**:
  1. `꾸러미.글` 매니페스트 파일 (이름, 버전, 의존성)
  2. 로컬 경로 기반 의존성 (처음에는 온라인 레지스트리 없이)
  3. `글도구 설치 경로` 명령
- **전제조건**: P2-1 (다중 파일 컴파일)

### P3: 완전 독립 (장기)

#### P3-1: 네이티브 백엔드 자체호스팅
- **현황**: 네이티브컴파일러.exe는 글cc(C 부트스트랩)로 빌드됨. 네이티브 백엔드로 자기 자신을 빌드하지 못함.
- **목표**: `네이티브컴파일러.exe`가 자기 자신의 소스를 컴파일하여 동일한 .exe를 생성
- **전제조건**:
  - P1-1 (구조체) — 컴파일러 내부의 SoA 패턴 대체 또는 배열 기반 유지
  - P1-2 (qword 배열) — 컴파일러의 풀 배열 접근
  - 문자열 처리 — 파서/렉서의 문자열 비교, 복사 등
  - 파일 I/O — 소스 파일 읽기, PE 파일 쓰기
- **마일스톤**:
  1. 단순 프로그램 컴파일 가능 (현재 달성)
  2. 렉서만 네이티브 컴파일 가능
  3. 파서 + 렉서 컴파일 가능
  4. 전체 컴파일러 컴파일 가능 (자체호스팅)
  5. SHA256 고정점 도달
- **예상 작업량**: 최상 (수 개월)

#### P3-2: Linux/macOS 지원
- **현황**: Windows PE 포맷만 지원
- **목표**: ELF (Linux) 및 Mach-O (macOS) 바이너리 생성
- **설계 방향**:
  1. 기계어 인코더는 공유 (x86-64 동일)
  2. PE 빌더를 추상화하여 ELF/Mach-O 빌더 추가
  3. 시스템 콜: Linux는 syscall, macOS는 libSystem
  4. 임포트: Linux는 .so 동적 링크, macOS는 .dylib
- **x86-64 기계어는 OS에 무관** — 바이너리 포맷과 시스템 호출만 다름
- **예상 작업량**: 상 (ELF는 PE보다 단순하므로 첫 번째 대상)

#### P3-3: 디버그 정보 생성
- **현황**: 없음. 크래시 시 디버거 사용 불가.
- **목표**: .pdb (Windows) 또는 DWARF (Linux) 디버그 정보 생성
- **최소 구현**: 함수명 + 줄 번호 매핑만으로도 디버거에서 스택 트레이스 가능
- **예상 작업량**: 중

#### P3-4: C 런타임 완전 제거
- **현황**: msvcrt.dll의 printf, malloc 등 31개 함수 사용
- **목표**: OS 시스템 콜만으로 모든 기능 구현
- **설계 방향**:
  1. 메모리: `VirtualAlloc`/`VirtualFree` (kernel32.dll) → 자체 malloc 구현
  2. 출력: `WriteConsoleA`/`WriteFile` (kernel32.dll) → 자체 printf 구현
  3. 파일: `CreateFileA`/`ReadFile`/`WriteFile` (kernel32.dll)
  4. 수학: 소프트웨어 구현 또는 x87/SSE 직접 사용
- **kernel32.dll만 의존** → C 런타임 완전 제거
- **예상 작업량**: 최상

## 비전

```
현재:   .글 → 글 컴파일러 → x86-64 PE → Windows 실행
        (msvcrt.dll + kernel32.dll 의존)

중기:   .글 → 글 컴파일러(글로 작성, 네이티브 빌드) → PE
        + 글 표준라이브러리 + LSP + 패키지 관리

장기:   .글 → 글 컴파일러 → PE/ELF/Mach-O
        + kernel32/libc 최소 의존
        + 자체 malloc/printf
        + Windows/Linux/macOS
```

**최종 목표**: 글 언어로 작성된 글 컴파일러가, 글 소스를 읽어, 어떤 외부 도구 없이, 주요 3개 운영체제에서 실행되는 네이티브 바이너리를 직접 생성하는 것.
